# AGENTS.md

## 项目概况

- **项目名**：ZZZ白话讲AI（ZZZ-Simple-AI）
- **目标**：用白话、极简、可复述的方式，帮零基础读者建立 AI 的基本认知
- **写给谁**：零基础职场人士、大学生，以及想先建立整体框架的读者

---

## 核心理念：三种学习方法贯穿始终

这个项目不只是讲 AI 知识，更是用三种经典学习方法来组织内容：

### 1. 费曼学习法（Feynman Technique）

**核心：用更简单的话重讲，逼自己确认是否真的理解。**

- 如果一段内容不能用大白话重讲一遍，说明还没写透
- 每一章、每一节，都要让读者能用自己的话复述出来
- 术语不是用来炫技的，是用来准确表达的；如果不能用大白话解释清楚，说明自己也没懂

**写作检验**：写完一段，问自己——如果一个零基础的朋友坐在我面前，我能用这段话让他听懂吗？

### 2. 苏格拉底式提问（Socratic Questioning）

**核心：用问题引导思考，而不是直接给答案。**

- 每一章、每一节，都从一个"你能理解的问题"开始
- 不要用术语开头，先回到原始问题
- 让读者先看到"为什么需要这个"，再看到"这是什么"

| 不要这样写 | 要这样写 |
|------------|----------|
| "RAG 是一种检索增强生成技术..." | "大模型明明学了很多，为什么答不上公司的新制度？" |
| "Agent 是一种能够自主决策的 AI 系统..." | "AI 为什么不会一直停在聊天框里？" |

### 3. 第一性原理（First Principles）

**核心：不从现成说法出发，而是先回到最原始的问题和约束。**

- 每讲一个概念，先问：它最初是为了解决什么问题？
- 不盲从行业术语，而是拆解到不能再拆的基本事实
- 让读者看到：这个技术不是凭空出现的，是从某个真实痛点长出来的

---

## 怎么讲清楚：三个原则

### 1. 先问"为什么会出现"

结合苏格拉底式提问，每一章、每一节，都要让读者先看到问题，再看到答案。

### 2. 让读者能复述

结合费曼学习法，写完一段，问自己：零基础读者能用自己的话讲出来吗？

- 如果一段内容不能用更简单的话重讲，说明还没写透
- 如果读者需要查词典才能继续，说明术语用早了
- 例子优先用职场、校园、日常场景，少用底层工程术语

### 3. 四个锚点不能丢

结合第一性原理，所有内容都要能回到这四个锚点：

| 锚点 | 一句话理解 |
|------|------------|
| **本质一**：概率模型 | 大模型最底层做的事，是根据前文预测下一个更可能出现的词 |
| **本质二**：上下文（Context） | 后面很多技术，本质上都在补充、整理和管理模型当前能看到的信息 |
| **主线一**：大模型派 | 更关心模型本身能不能继续变强 |
| **主线二**：驾驭工程派 | 更关心怎样用好模型的能力，让它在真实场景里靠谱地跑起来 |

每一章都要说清楚：这一章更靠近哪条主线，以及它如何回到两个本质。

## 具体怎么写：章节模板

### 章节开头（2 块）

**1. 读前先想**

用 1-2 个短问题把读者拉回原始问题。问题要口语化、有引导感。

| 原来 | 改进 |
|------|------|
| 为什么大模型第一次走进大众视野，是以"网页聊天机器人"的形式出现的？ | 为什么 AI 的第一副面孔是"聊天框"？ |
| 为什么只靠模型参数里的知识，很快就不够用了？ | 大模型明明"学富五车"，为什么却答不上公司的新制度？ |

**2. 章节定位**

用表格说明：

| 维度 | 内容 |
|------|------|
| 横向定位 | 这一章在整个项目中的位置 |
| 本章回扣 | 1. 如何回到本质一（概率模型）<br>2. 如何回到本质二（上下文） |

### 小节结构（6 块）

每个小节**默认**包含：

1. **提问**：1-2 个精炼问题，聚焦核心疑惑
2. **术语**：中文、英文、（简称）、补充说明
3. **一句话解释**：最简版本，方便复述
4. **你可以先这样理解**：白话展开，带场景感
5. **示例**：具体场景，帮助代入
6. **你只需要记住**：2-3 条核心要点

**表达建议：**

- 用口语替代模板："说白了"、"其实"、"一句话"
- 段落长短不一，有呼吸感
- 允许有作者的声音，不要像填空

---

### 小节模板灵活使用规则

模板是工具，不是枷锁。当模板让内容显得冗余时，就简化它。

| 场景 | 模板使用建议 |
| :--- | :--- |
| **核心概念首次出现** | 完整使用6块模板 |
| **对比/归类/总结性小节** | 可以简化，保留"一句话金句+表格+总结" |
| **承上启下的过渡小节** | 可以只用"过渡句+核心内容" |
| **章节小结（本章要义）** | 必须使用统一四段式格式 |

**简化示例（归类/总结性小节）**：
```markdown
💭 **上一节我们知道了**：...（过渡句）

## 📖 第X小节：XXXX——XXXXX

> 💡 **一句话理解**：...（金句放在引用块中）

### 📊 对照表
| 项目 | 说明 |
| :--- | :--- |
| ... | ... |

这样看的好处：...（一句总结，替代"你只需要记住"）
```

### 叙事连贯性规范（必读）

好的技术内容不是知识点的堆砌，而是一场有逻辑的"思维旅程"。这三条规范确保读者能顺畅地跟随你的思路，而不是被突然抛来抛去。

---

### 1. 章与章之间：自然衔接，铺垫下文

**每一章的结尾，必须抛出一个疑问，引向下一张。**

这不是可有可无的装饰，而是叙事连贯的核心。读者读完一章后，应该自然产生"那接下来呢？"的好奇心。

**写法：**
- 从本章的核心问题，自然延伸出下一章要解决的问题
- 不要生硬预告标题，而是抛出下一章要回答的"疑问"
- 让读者觉得"原来还有这层"，而不是"又要讲新东西了"

| ❌ 不好的写法 | ✅ 好的写法 |
|------------|------------|
| "下一章我们将介绍 AI 应用是如何发展到 Agent 的。" | "明白了大模型只是'接话'和'看纸条'，下一个问题自然就来了：既然它只会接话，那它是怎么从'聊天框'一步步进化到'能做事'的？" |

**关键区别**：好的衔接是从读者的认知缺口中自然长出来的。

**正文用词规范**：
- ❌ 正文中不要使用"钩子"一词（这是给作者看的内部术语）
- ✅ 使用"为下一章铺垫"、"引出下一个问题"等自然表达

---

### 2. 小节与小节之间：层层递进，螺旋上升

**章节内部的每一个小节，必须层层递进、互相连接。**

读者不是在读六个孤立的小节，而是在走一条有起承转合的路。每一小节都应该：

- 🔗 **承接上文**：开头用一句话回顾上一小节的核心结论
- 🎯 **聚焦新问题**：抛出本小节要回答的具体问题
- 💡 **给出答案**：用白话解释清楚
- 🌉 **铺垫下文**：结尾为下一小节埋下伏笔或自然过渡

**过渡句规范**：
- 使用 💭 标识过渡句
- 格式：`💭 **上一节我们知道了/刚才我们看了**：...（回顾核心结论）...（抛出本小节问题）`

**递进关系示例（以第1章为例）**：
```
第1小节：本质——按概率生成词元
    ↓ 抛出：它生成得好不好，取决于什么？
第2小节：上下文——模型能看到的全部信息  
    ↓ 抛出：前两者是什么关系？为什么上下文这么重要？
第3小节：串联起来看——本质 + 上下文 = 精准输出
    ↓ 汇总本章，引出下一章的问题
```

**检验标准**：跳过任意一个小节，读者会不会觉得"突然断了"？

---

### 3. 章末汇总：把散落的珍珠串成项链

**每一章的结束，必须有一个汇总性质的小节。**

不要让读者自己总结——帮他们提炼。汇总小节的作用是：

- 🎯 **回扣锚点**：明确本章如何回到"两个本质"
- 🔗 **串联线索**：用一句话串起本章所有小节的核心脉络
- 📊 **可视化总结**：用表格或清单呈现核心要点
- 🌉 **铺垫下文**：自然抛出下一章要解答的问题

**汇总小节的结构（标准四段式）**：
```markdown
## 📝 本章要义

> 💡 **一句话核心结论**（金句式）

---

### 🔗 核心脉络回顾
| 小节 | 关键认知 | 递进关系 |
| :--- | :--- | :--- |
| 📖 第1节 | ... | 回答了"是什么" |
| 📖 第2节 | ... | 回答了"靠什么" |
| 📖 第3节 | ... | 回答了"为什么重要" |

---

### 🎯 回到两个本质
| 本质 | 核心理解 |
| :--- | :--- |
| 🎯 **本质一** | ... |
| 📋 **本质二** | ... |

---

### 🌉 为下一章铺垫
明白了这些，下一个问题自然就来了：...
```

**格式说明**：
- 使用 `---` 分隔线增强层次感
- 小节标题统一用 📖 标识
- 两个本质用表格呈现（比列表更清晰）
- 正文使用"为下一章铺垫"，避免使用"钩子"一词

## 视觉与排版规范

好的内容不仅要有逻辑，还要有"阅读友好性"。

### 表情符号使用

适当使用表情符号（emoji）可以提升视觉层次和阅读体验：

| 使用场景 | 示例 | 说明 |
|---------|------|------|
| 章节标题 | `# 🤖 第一章` | 主题级标识 |
| 读前先想 | `## 🤔 读前先想` | 让读者快速识别内容类型 |
| 章节定位 | `## 🧭 章节定位` | 结构导航 |
| 小节标题 | `## 📖 第X小节` | 统一标识 |
| 提问 | `### ❓ 提问` | 引发思考的问题 |
| 术语 | `### 📝 术语` | 专业概念定义 |
| 一句话解释 | `### 💬 一句话解释` | 核心概念提炼 |
| 深入理解 | `### 🧐 你可以先这样理解` | 白话展开 |
| 示例 | `### 📚 示例` | 具体场景 |
| 核心要点 | `### ✅ 你只需要记住` | 重点总结 |
| 过渡句 | `💭 **上一节我们知道了**` | 小节间衔接 |
| 金句 | `> 💡 **一句话核心结论**` | 引用块中强调 |
| 列表项 | `- 💼 职场新人` | 增加视觉引导 |
| 对比表格 | `❌ 不要这样写` / `✅ 要这样写` | 直观表达好坏对比 |
| 核心脉络 | `### 🔗 核心脉络回顾` | 章末结构 |
| 本质回扣 | `### 🎯 回到两个本质` | 章末结构 |
| 铺垫下文 | `### 🌉 为下一章铺垫` | 章末结构 |

**注意事项：**
- 不要过度使用，每段1-2个即可
- 同一类型的内容使用一致的 emoji
- 优先选择通用易懂的符号
- 章末汇总（本章要义）的四个部分使用统一的 emoji 标识：📝（标题）、🔗（脉络）、🎯（本质）、🌉（铺垫）

### 排版技巧

- **引用块（>）**：用于突出核心理念或金句
- **表格**：用于对比、分类、速览，减少大段文字
- **代码块**：用于示例对话，区分于正文
- **加粗**：用于术语首次出现、核心结论
- **分隔线（---）**：用于区分大章节

### 层次结构

```
# 一级标题（文章标题）

## 二级标题（大章节）

### 三级标题（小节）

#### 四级标题（小节内的细分，尽量少用）
```

保持层次清晰，不要让读者"迷路"。

## 协作约定

- 用户口述想法时，先整理成结构化要点，再压缩成简化表达
- 如果用户表述不准确，主动纠正，不要为了顺从而保留错误说法
- 发现内容偏离"简单、可复述、贴近工作场景"时，主动收窄范围
