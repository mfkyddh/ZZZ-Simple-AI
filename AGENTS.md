# AGENTS.md

## 项目概况

- **项目名**：ZZZ白话讲AI（ZZZ-Simple-AI）
- **目标**：用白话、极简、可复述的方式，帮零基础读者建立 AI 的基本认知
- **写给谁**：零基础职场人士、大学生，以及想先建立整体框架的读者

**核心原则**：一切内容都服务于零基础读者。他们不需要知道所有细节，只需要能听懂、能复述、能用上。

---

## 核心理念：三个学习方法

| 方法 | 核心 | 写作检验 |
|:---|:---|:---|
| **费曼学习法** | 用更简单的话重讲 | 一个零基础的朋友坐在我面前，我能让他听懂吗？ |
| **苏格拉底式提问** | 用问题引导思考 | 不用术语开头，先回到原始问题 |
| **第一性原理** | 回到最原始的问题和约束 | 这个概念最初是为了解决什么问题？ |

**四个锚点**（所有内容都要能回到这里）：

| 锚点 | 一句话理解 |
|:---|:---|
| 🎯 **本质一**：概率模型 | 大模型最底层做的事，是根据前文预测下一个更可能出现的词 |
| 📋 **本质二**：上下文（Context） | 后面很多技术，本质上都在补充、整理和管理模型当前能看到的信息 |
| **主线一**：大模型派 | 更关心模型本身能不能继续变强 |
| **主线二**：驾驭工程派 | 更关心怎样用好模型的能力，让它在真实场景里靠谱地跑起来 |

## 写作红线（从错误中学习）

### ❌ 不要啰嗦

**错误示范**：第五章初稿8个小节，逐部件对比，十几张表格，4500字。

**问题**：初学者被信息淹没，记不住核心观点。

**正确做法**：
- 一章最多3-4个小节
- 每个小节只说一个核心观点
- 表格最多1-2张，能用文字说清就不用表格

---

### ❌ 不要自嗨

**错误示范**：详细解释"第一性原理"、"核心矛盾"、"驾驭工程"等理论框架。

**问题**：读者要的是"这是什么"，不是"我是怎么分析的"。

**正确做法**：
- 理论留给自己，白话留给读者
- 不要用"主要矛盾""次要矛盾"等分析术语
- 让读者通过故事和例子自己领悟

---

### ❌ 不要炫技

**错误示范**：复杂的流程图、架构图、对照表。

**问题**：形式大于内容，核心观点被淹没。

**正确做法**：
- 一个简单比喻胜过一张复杂图表
- 户型图、瑞士军刀、机器人——用生活化比喻
- 让读者能复述，而不是能看图

---

### ❌ 不要忘记受众

**核心提醒**：读者是零基础，不是同行。

| 不要这样写 | 要这样写 |
|:---|:---|
| "六部件框架的透视法应用" | "学会这套方法，看透任何AI产品" |
| "逐部件对比分析" | "它们都有这六个东西，只是用法不同" |
| "场景决定产品形态" | "同样的工具，不同的人用，效果不同" |

## 章节结构标准

### 章节开头（2块）

**1. 读前先想**

用1-2个口语化问题拉回原始问题：
- ❌ "为什么大模型第一次走进大众视野..."
- ✅ "为什么AI的第一副面孔是聊天框？"

**2. 章节定位**

```markdown
| 维度 | 内容 |
|:---|:---|
| 横向定位 | 这一章在整个项目中的位置 |
| 本章回扣 | 1. 如何回到本质一（概率模型）<br>2. 如何回到本质二（上下文） |
```

---

### 小节结构（6块模板）

每个小节默认包含：

1. **提问**：1-2个精炼问题，聚焦核心疑惑
2. **术语**：中文、英文（缩写）
3. **一句话解释**：最简版本，方便复述
4. **你可以先这样理解**：白话展开，带场景感
5. **示例**：具体场景，帮助代入
6. **你只需要记住**：2-3条核心要点

**灵活使用**：
- 核心概念首次出现：完整使用6块
- 对比/归类/总结性小节：保留"一句话+表格+总结"
- 承上启下的过渡小节：只用"过渡句+核心内容"

---

### 叙事连贯

**小节之间**：用 💭 过渡句承接
```markdown
💭 **上一节我们知道了**：...（回顾）...（抛出本小节问题）
```

**章末结构（本章要义）**：
1. 一句话核心结论（金句）
2. 核心脉络回顾（表格）
3. 回到两个本质（表格）
4. 为下一章铺垫（抛出疑问）

## 表达规范

### 中文书面语（强制性）

| ❌ 禁用 | ✅ 规范 |
|:---|:---|
| 怎么接 | 如何接收/连接 |
| 稳不稳/靠不靠谱 | 是否稳定/可靠 |
| 兜底 | 风险防控/备选方案 |
| 放手做 | 充分授权 |
| 搞砸 | 造成错误/引发问题 |
| 啰嗦 | 交互步骤较多 |
| 变轻/变重/留下来 | 简化/保留/长期存在 |
| 站队/吃掉/钩子 | 偏向/取代/铺垫 |

| 提问 | `### ❓ 提问` |

| 一句话解释 | `### 💬 一句话解释` |

### 排版技巧

- **引用块（>）**：突出核心理念或金句
- **表格**：用于对比、分类，减少大段文字
- **代码块**：示例对话
- **加粗**：术语首次出现、核心结论
- **分隔线（---）**：区分大章节

## 写作检查清单

写完一章，问自己：

- [ ] 读者能用自己的话复述这一章的核心观点吗？（费曼检验）
- [ ] 这一章是从一个真实问题开始的吗？（苏格拉底检验）
- [ ] 这一章和四个锚点的关系说清楚了吗？（第一性原理检验）
- [ ] 有没有超过3-4个小节？（精简检验）
- [ ] 有没有用大段理论解释？（去自嗨检验）
- [ ] 表格和图示是否必要？（去炫技检验）
- [ ] 例子是职场/校园场景吗？（受众检验）
- [ ] 章节末尾有铺垫下文吗？（连贯检验）

## 协作约定

1. **先理解，再写作**：用户口述想法时，先整理成结构化要点
2. **敢纠正**：如果用户表述不准确，主动纠正，不要为了顺从而保留错误说法
3. **主动收窄**：发现内容偏离"简单、可复述、贴近工作场景"时，主动提出简化方案
4. **先问后写**：不确定时先问清楚，不要基于猜测写大段内容
