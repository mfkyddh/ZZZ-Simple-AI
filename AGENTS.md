# AGENTS.md

## 项目概况

- **项目名**：ZZZ白话讲AI（ZZZ-Simple-AI）
- **目标**：用白话、极简、可复述的方式，帮零基础读者建立 AI 的基本认知
- **写给谁**：零基础职场人士、大学生，以及想先建立整体框架的读者

---

## 怎么讲清楚：三个原则

### 1. 先问"为什么会出现"

不要用术语开头，先回到原始问题。

| 不要这样写 | 要这样写 |
|------------|----------|
| "RAG 是一种检索增强生成技术..." | "大模型明明学了很多，为什么答不上公司的新制度？" |
| "Agent 是一种能够自主决策的 AI 系统..." | "AI 为什么不会一直停在聊天框里？" |

每一章、每一节，都要让读者先看到问题，再看到答案。

### 2. 让读者能复述

写完一段，问自己：零基础读者能用自己的话讲出来吗？

- 如果一段内容不能用更简单的话重讲，说明还没写透
- 如果读者需要查词典才能继续，说明术语用早了
- 例子优先用职场、校园、日常场景，少用底层工程术语

### 3. 四个锚点不能丢

所有内容都要能回到这四个锚点：

| 锚点 | 一句话理解 |
|------|------------|
| **本质一**：概率模型 | 大模型最底层做的事，是根据前文预测下一个更可能出现的词 |
| **本质二**：上下文（Context） | 后面很多技术，本质上都在补充、整理和管理模型当前能看到的信息 |
| **主线一**：大模型派 | 更关心模型本身能不能继续变强 |
| **主线二**：驾驭工程派 | 更关心怎样把模型能力接住、管住、组织成稳定系统 |

每一章都要说清楚：这一章更靠近哪条主线，以及它如何回到两个本质。

## 具体怎么写：章节模板

### 章节开头（2 块）

**1. 读前先想**

用 1-2 个短问题把读者拉回原始问题。问题要口语化、有引导感。

| 原来 | 改进 |
|------|------|
| 为什么大模型第一次走进大众视野，是以"网页聊天机器人"的形式出现的？ | 为什么 AI 的第一副面孔是"聊天框"？ |
| 为什么只靠模型参数里的知识，很快就不够用了？ | 大模型明明"学富五车"，为什么却答不上公司的新制度？ |

**2. 章节定位**

用表格说明：

| 维度 | 内容 |
|------|------|
| 横向定位 | 这一章在整个项目中的位置 |
| 本章回扣 | 1. 如何回到本质一（概率模型）<br>2. 如何回到本质二（上下文） |

### 小节结构（6 块）

每个小节默认包含：

1. **提问**：1-2 个精炼问题，聚焦核心疑惑
2. **术语**：中文、英文、（简称）、补充说明
3. **一句话解释**：最简版本，方便复述
4. **你可以先这样理解**：白话展开，带场景感
5. **示例**：具体场景，帮助代入
6. **你只需要记住**：2-3 条核心要点

**表达建议：**

- 用口语替代模板："说白了"、"其实"、"一句话"
- 段落长短不一，有呼吸感
- 允许有作者的声音，不要像填空

## 写作检查清单

写完一章，问自己：

- [ ] 读者能复述这一章的核心观点吗？
- [ ] 这一章和四个锚点的关系说清楚了吗？
- [ ] 提问够精炼吗？（每节 1-2 个）
- [ ] 有没有术语没解释就直接用了？
- [ ] 例子是职场/校园场景吗？

## 仓库结构

| 路径 | 作用 |
|------|------|
| `README.md` | 项目简介、阅读入口 |
| `docs/` | 知识库正文，按顺序编号 |
| `indexes/` | 术语表等索引 |
| `site/` | 预留：HTML 或静态站点产物 |
| `changes/` | 预留：变更记录 |

Markdown 是唯一长期内容源，HTML 属于派生产物。

---

## 术语规范

- 第一次出现用 `中文（English）` 或 `中文（缩写）` 格式
- 仓库中的 `indexes/术语表.md` 是统一口径
- 简称只在确实存在常用简称时出现，不要写"简称：无"

---

## 协作约定

- 用户口述想法时，先整理成结构化要点，再压缩成简化表达
- 如果用户表述不准确，主动纠正，不要为了顺从而保留错误说法
- 发现内容偏离"简单、可复述、贴近工作场景"时，主动收窄范围
