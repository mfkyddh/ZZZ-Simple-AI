# AGENTS.md

## 项目概况

- 项目中文名：`ZZZ白话讲AI`
- 项目英文名：`ZZZ-Simple-AI`
- 项目根目录：`D:\Projects\ZZZ-Simple-AI`
- 本项目是一个面向零基础职场人士和大学生的 AI 入门知识库。
- 项目目标是用白话、极简、可复述的方式，帮助读者建立 AI 的基本认知和使用框架。

## 面向人群优先原则

- 后续所有项目内容，都必须优先从零基础职场人士和大学生的视角来思考。
- 写作时优先考虑“他们能不能快速看懂、能不能复述、能不能立刻联想到自己的学习或工作场景”。
- 当“技术完整性”和“目标人群可理解性”发生冲突时，在不失真的前提下，优先选择更容易理解的表达。
- 例子优先使用职场、校园、日常学习和常见办公场景，不优先使用底层研发、训练框架和工程基础设施场景。
- 任何章节在生成前，都应先自检：这段内容是不是在帮目标人群理解，而不是在向懂行的人证明自己专业。

## 第一章主线约束

- 后续所有章节、所有小节，都必须反复回到第一章的两个核心点。
- 核心点一：大模型本质上是一个生成文字的概率模型。
- 核心点二：后续很多技术和产品形态，本质上都是在补充、整理和管理模型上下文（Context）。
- 如果某一节讲着讲着脱离了这两个核心点，必须主动收回来，否则内容会越写越散。

## 项目结构主线

- 本项目当前只保留一条整体结构主线：横向主线。
- 横向主线用于解释业界两种主要思路：
  - 大模型派（Big Model）
  - 外部编排派（Harness Engineering，项目内术语）
- 外部编排派（Harness Engineering，项目内术语）在本项目中，指的是通过模型外部的上下文管理、工具调用、工作流、记忆、约束和执行系统来增强智能体（Agent）能力的思路。
- 本项目当前不采用“五层纵向主线”作为整体结构，不在总纲层面继续展开能源、芯片、基础设施、模型、应用这类分层叙事。

## 当前工作方式

### 阶段 1：构建知识库大纲

- 采用“苏格拉底提问式”推进。
- 一次只推进一个问题、一个章节或一个小段落。
- 先确定章节名称，再根据用户口述的零散想法整理成简化版内容。
- 每轮输出保持短小，先确认，再继续扩展。
- 未经用户确认，不批量生成多个章节正文。

### 阶段 2：技术沉淀与维护

- 知识库初步稳定后，再生成 HTML。
- HTML 生成链路稳定后，再沉淀脚本、目录结构和维护流程。
- 后续按“更新知识库 -> 生成 HTML -> 再更新知识库”的循环持续维护。

## 内容原则

- 以“简化、简单、讲明白”为最高优先级。
- 默认从零基础职场人士或大学生的视角写作。
- 后续所有新增、重写、扩写内容，都必须先围绕目标人群的理解成本来取舍。
- 强调作者个人判断与思考，不追求百科式完整。
- 一章只回答一个核心问题。
- 章节先锁定骨架，再补正文，避免写成连续散文式堆叠。
- 每一节只回答一个问题，避免不同小节反复讲同一件事。
- 章节默认采用固定结构化模板，避免自由散文式写法，方便后续生成网页。
- 每一章都应尽量说明它在横向主线中的位置，避免读者不知道这一章在整个项目里扮演什么角色。
- 先讲“这是什么”和“为什么重要”，再讲术语或细节。
- 优先使用白话、短句、短段落和现实例子。
- 遇到明显的流程、调用链、输入输出关系或阶段演进时，优先用 Mermaid 流程图表达，不用大段文字重复解释。
- 遇到明显的对照、分类、阶段差异、优缺点或角色分工时，优先用 Markdown 表格表达。
- 图表和表格不是装饰，它们应直接承担解释任务，用来减少正文长度、提高扫读效率。
- 图表默认保持极简，一图只讲一件事，一表只解决一个对照问题。
- 不主动展开底层算法、复杂数学推导、论文细节和工程实现细枝末节。
- 不为了显得专业而堆砌术语。
- 如果某个概念能讲得更简单，优先选择更简单的表达。
- 专业术语第一次出现时，优先同时给出中文名、英文词和一句最简白话解释。

## 协作约束

- 默认使用中文沟通和写作。
- 用户口述想法时，先整理为结构化要点，再压缩成简化表达。
- 写作或改写前，先判断这段内容对零基础职场人士和大学生是否真的有帮助。
- 如果用户口述中的术语、产品表述、技术归因或历史发展不够准确，必须主动纠正，不能为了顺着用户表达而保留错误说法。
- 术语写法默认使用 `中文（English）` 这种形式，不单独裸写英文单词；缩写可放在括号或术语项中。
- 不要在正文还没有正式抛出、解释某个概念之前，就在更前面的标题、定位条、总结句或判断句里直接使用它。
- 任何新概念都应先定义、再引用；如果读者此时还不该知道这个词，就不要提前让它出现。
- 发现内容偏离“简单、可复述、贴近工作场景”时，应主动收窄范围。
- 需要引用时效性强的信息时，先核实再写入。
- 不要把内容写成百科、论文摘要或底层原理导论。

## 仓库约定

- `README.md`：项目简介、目标用户、工作方式。
- `docs/`：后续知识库正文，采用扁平化结构，不在 `docs/` 下继续分模块子目录。
- `docs/` 下的文章用文件名前缀体现顺序，例如 `01-什么是大模型.md`。
- 新增或重写章节时，默认延续 `docs/` 扁平化结构，不单独新建模块文件夹。
- `indexes/`：目录、标签、术语等索引。
- `changes/`：变更记录。
- `site/`：HTML 或静态站点产物。
- Markdown 是唯一长期内容源；HTML 属于派生产物。

## 通用章节模板

- 后续章节默认采用固定 `5` 块模板。
- 每章标题下方默认增加一个单独的 `章节定位` 模块，不再用两行口语化提示代替。
- `章节定位` 默认使用 Markdown 表格，至少包含两项：
  - `横向定位`
  - `本章回扣`
- `本章回扣` 默认用编号句式呈现，避免写成松散长句。
- 如果需要补充章节导语，优先使用单独的 `本章说明` 模块，不要直接写成连续口语段落。
- 章节的核心解释块仍以 `术语`、`一句话解释`、`你可以先这样理解`、`示例`、`你只需要记住` 为基础。
- 如果某一节存在明显流程，优先补一个 `流程图` 模块；流程图默认使用 Markdown 中的 Mermaid 代码块。
- 如果某一节存在明显对照关系，优先补一个 `表格` 或 `对照表` 模块；表格默认使用 Markdown 原生表格。
- 一章如果存在明显的流程或对照关系，默认至少加入一个图或表，不继续只靠正文硬讲。
- 模板顺序默认如下：
  1. `章节定位`
  2. `本章说明`（按需使用）
  3. `术语`
  4. `一句话解释`
  5. `你可以先这样理解`
  6. `示例`
  7. `你只需要记住`
- 如无必要，不新增第 `6` 块及以上的自由段落。
- 如果某一节确实需要补充，可以额外增加 `常见误区`，但不作为默认必选项。
- 每一块内容都要短，方便后续直接映射为网页组件。
