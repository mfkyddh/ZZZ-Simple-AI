# AGENTS.md

## 项目概况

- 项目中文名：`ZZZ白话讲AI`
- 项目英文名：`ZZZ-Simple-AI`
- 项目根目录：`D:\Projects\ZZZ-Simple-AI`
- 本项目是一个面向零基础职场人士和大学生的 AI 入门知识库。
- 项目目标是用白话、极简、可复述的方式，帮助读者建立 AI 的基本认知和使用框架。

## 面向人群优先原则

- 后续所有项目内容，都必须优先从零基础职场人士和大学生的视角来思考。
- 写作时优先考虑“他们能不能快速看懂、能不能复述、能不能立刻联想到自己的学习或工作场景”。
- 当“技术完整性”和“目标人群可理解性”发生冲突时，在不失真的前提下，优先选择更容易理解的表达。
- 例子优先使用职场、校园、日常学习和常见办公场景，不优先使用底层研发、训练框架和工程基础设施场景。
- 任何章节在生成前，都应先自检：这段内容是不是在帮目标人群理解，而不是在向懂行的人证明自己专业。

## 解释方法原则

- 后续所有章节和小节，默认同时使用费曼学习法（Feynman Technique）和第一性原理（First Principles）。
- 费曼学习法（Feynman Technique）在本项目中的要求是：先把内容讲到零基础读者能复述，再决定是否保留术语和细节；如果一段内容不能用更简单的话重讲，说明这段内容还没有写透。
- 第一性原理（First Principles）在本项目中的要求是：不要从现成术语和行业黑话出发，而要先回到“原始问题是什么、旧办法哪里不够、所以为什么会出现这项技术或框架”。
- 后续所有章节和小节，默认采用“先提问，再解释；先问题，再术语；先本质，再框架”的表达顺序。
- 如果某一段内容无法回答“它为什么会出现、为什么要引入、为什么这里要用这个框架来看”，这一段就还没有写稳。

## 第一章主线约束

- 后续所有章节、所有小节，都必须反复回到第一章的两个核心点。
- 核心点一：大模型本质上是一个生成文字的概率模型。
- 核心点二：后续很多技术和产品形态，本质上都是在补充、整理和管理模型上下文（Context）。
- 如果某一节讲着讲着脱离了这两个核心点，必须主动收回来，否则内容会越写越散。

## 叙事结构约束

- 本项目后续所有 README、章节开头、目录说明和阶段性总结，都应优先突出四个锚点：
  - 两个本质：概率模型、模型上下文（Context）
  - 两条主线：大模型派（Big Model）、驾驭工程派（Harness Engineering）
- 两个本质是全项目的解释底座；两条主线是全项目的叙事骨架。
- 如果某一章不能说明自己与这四个锚点的关系，这一章就还没有写稳。
- README 和章节开头应尽量把这四个锚点放在更靠前的位置，而不是埋到后文。
- 后续新增章节时，应主动说明：这一章更靠近哪条主线，以及它如何回到第一章的两个本质。

## 项目结构主线

- 本项目当前只保留一条整体结构主线：横向主线。
- 横向主线用于解释业界两种主要思路：
  - 大模型派（Big Model）
  - 驾驭工程派（Harness Engineering，项目内术语）
- 驾驭工程派（Harness Engineering，项目内术语）在本项目中，指的是通过模型外部的上下文管理、工具调用、工作流、记忆、约束和执行系统来增强智能体（Agent）能力的思路。
- 本项目当前不采用“五层纵向主线”作为整体结构，不在总纲层面继续展开能源、芯片、基础设施、模型、应用这类分层叙事。

## 当前工作方式

### 阶段 1：构建知识库大纲

- 采用“苏格拉底提问式”推进。
- 一次只推进一个问题、一个章节或一个小段落。
- 先确定章节名称，再根据用户口述的零散想法整理成简化版内容。
- 每轮输出保持短小，先确认，再继续扩展。
- 未经用户确认，不批量生成多个章节正文。

### 阶段 2：技术沉淀与维护

- 知识库初步稳定后，再生成 HTML。
- HTML 生成链路稳定后，再沉淀脚本、目录结构和维护流程。
- 后续按“更新知识库 -> 生成 HTML -> 再更新知识库”的循环持续维护。

## 内容原则

- 以“简化、简单、讲明白”为最高优先级。
- 默认从零基础职场人士或大学生的视角写作。
- 后续所有新增、重写、扩写内容，都必须先围绕目标人群的理解成本来取舍。
- 后续所有新增、重写、扩写内容，都应先问“为什么会出现”，再讲“它是什么”。
- 强调作者个人判断与思考，不追求百科式完整。
- 一章只回答一个核心问题。
- 章节先锁定骨架，再补正文，避免写成连续散文式堆叠。
- 每一节只回答一个问题，避免不同小节反复讲同一件事。
- 章节默认采用固定结构化模板，避免自由散文式写法，方便后续生成网页。
- 写作时优先让读者先看到“两个本质”和“两条主线”，再进入具体术语和细节。
- 每一章都应尽量说明它在横向主线中的位置，避免读者不知道这一章在整个项目里扮演什么角色。
- 先讲“这是什么”和“为什么重要”，再讲术语或细节。
- 优先使用白话、短句、短段落和现实例子。
- 遇到明显的流程、调用链、输入输出关系或阶段演进时，优先用 Mermaid 流程图表达，不用大段文字重复解释。
- 遇到明显的对照、分类、阶段差异、优缺点或角色分工时，优先用 Markdown 表格表达。
- 图表和表格不是装饰，它们应直接承担解释任务，用来减少正文长度、提高扫读效率。
- 图表默认保持极简，一图只讲一件事，一表只解决一个对照问题。
- 不主动展开底层算法、复杂数学推导、论文细节和工程实现细枝末节。
- 不为了显得专业而堆砌术语。
- 如果某个概念能讲得更简单，优先选择更简单的表达。
- 专业术语第一次出现时，优先同时给出中文名、英文词和一句最简白话解释。
- 术语块中的字段按需出现；如果没有常用简称，就直接省略 `简称`，不要写 `简称：无`。
- 仓库中的 `indexes/术语表.md` 是项目术语的统一口径；新增术语、修正译名或统一简称时，应优先更新这份术语表，再同步正文。

## 协作约束

- 默认使用中文沟通和写作。
- 用户口述想法时，先整理为结构化要点，再压缩成简化表达。
- 写作或改写前，先判断这段内容对零基础职场人士和大学生是否真的有帮助。
- 如果用户口述中的术语、产品表述、技术归因或历史发展不够准确，必须主动纠正，不能为了顺着用户表达而保留错误说法。
- 术语写法默认使用 `中文（English）` 这种形式，不单独裸写英文单词；缩写可放在括号或术语项中。
- README、章节导语、表格和总结句中，专业术语第一次出现时，也必须遵守 `中文（English）` 或 `中文（缩写）` 的写法，不能只写英文或只写缩写。
- 如果正文用词与 `indexes/术语表.md` 不一致，应以术语表为准进行修正。
- 不要在正文还没有正式抛出、解释某个概念之前，就在更前面的标题、定位条、总结句或判断句里直接使用它。
- 任何新概念都应先定义、再引用；如果读者此时还不该知道这个词，就不要提前让它出现。
- 发现内容偏离“简单、可复述、贴近工作场景”时，应主动收窄范围。
- 需要引用时效性强的信息时，先核实再写入。
- 不要把内容写成百科、论文摘要或底层原理导论。

## 仓库约定

- `README.md`：项目简介、目标用户、工作方式。
- `docs/`：后续知识库正文，采用扁平化结构，不在 `docs/` 下继续分模块子目录。
- `docs/` 下的文章用文件名前缀体现顺序，例如 `01-什么是大模型.md`。
- 新增或重写章节时，默认延续 `docs/` 扁平化结构，不单独新建模块文件夹。
- `indexes/`：目录、标签、术语等索引。
- `changes/`：变更记录。
- `site/`：HTML 或静态站点产物。
- Markdown 是唯一长期内容源；HTML 属于派生产物。

## 通用章节模板

- 后续章节默认采用“章节级 2 块 + 小节级 6 块”的固定模板。
- 每章标题下方默认先增加一个单独的 `读前先想` 模块，再进入 `章节定位`。
- `读前先想` 的作用，是用短问题把读者带回原始问题，让读者先产生阅读兴趣，再进入正文。
- `读前先想` 默认使用 `2` 个问题，最多不超过 `3` 个；每个问题都必须是白话、单句、短句，不写解释性长句。
- `章节定位` 默认使用 Markdown 表格，至少包含两项：
  - `横向定位`
  - `本章回扣`
- `本章回扣` 默认用编号句式呈现，避免写成松散长句。
- 每一个小节在原五段式卡片前，默认新增一个 `提问` 模块。
- `提问` 模块的作用，是先用第一性原理把读者带回原始问题；默认至少回答三件事：
  - 为什么会出现这项技术、概念或分法
  - 为什么要引入这项技术、概念或分法
  - 为什么这里要用这个框架来看它
- 如果某一节不是在讲“技术”，而是在讲“概念”或“分法”，`提问` 的第三问可以改写成更自然的白话，但作用不变。
- 小节的核心解释块仍以 `术语`、`一句话解释`、`你可以先这样理解`、`示例`、`你只需要记住` 为基础。
- `术语` 模块默认包含 `中文`、`英文` 和必要的补充说明；`简称` 只在确实存在常用简称时出现。
- 如果某一节存在明显流程，优先补一个 `流程图` 模块；流程图默认使用 Markdown 中的 Mermaid 代码块。
- 如果某一节存在明显对照关系，优先补一个 `表格` 或 `对照表` 模块；表格默认使用 Markdown 原生表格。
- 一章如果存在明显的流程或对照关系，默认至少加入一个图或表，不继续只靠正文硬讲。
- 模板顺序默认如下：
  - 章节级：
    1. `读前先想`
    2. `章节定位`
  - 小节级：
    1. `提问`
    2. `术语`
    3. `一句话解释`
    4. `你可以先这样理解`
    5. `示例`
    6. `你只需要记住`
- 如无必要，不新增第 `6` 块及以上的自由段落。
- 如果某一节确实需要补充，可以额外增加 `常见误区`，但不作为默认必选项。
- 每一块内容都要短，方便后续直接映射为网页组件。
