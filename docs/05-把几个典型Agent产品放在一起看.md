# 第五章 把 Openclaw、龙虾、Cloud Code 放在一起看

## 读前先想

- 同样都叫智能体（Agent）产品，为什么用起来差这么多？
- 真正拉开差距的，到底是模型名字，还是模型上下文（Context）、工具和权限？

## 章节定位

| 维度 | 内容 |
| --- | --- |
| 横向定位 | 这一章继续站在驾驭工程派（Harness Engineering）的视角，比较不同产品是怎么组织模型上下文（Context）、工具和执行流程的。 |
| 本章回扣 | 1. 这几类产品底层仍然依赖同一个概率模型逻辑。<br>2. 它们体验上的差异，很多来自模型上下文（Context）组织方式、工具系统和执行约束，而不只是模型名字不同。 |

## 先看对比框架

| 比较维度 | 后续会重点看什么 |
| --- | --- |
| 目标入口 | 用户是怎么下达任务的 |
| 模型上下文（Context）来源 | 它能看到哪些信息，哪些看不到 |
| 工具范围 | 它能调哪些工具，动作边界在哪里 |
| 执行循环 | 它是一轮一轮推进，还是一次性输出 |
| 权限与确认 | 哪些动作需要用户同意 |
| 可复用能力 | 技能（Skill）、模板、预设流程怎么组织 |
| 扩展方式 | 模型上下文协议（Model Context Protocol，`MCP`）或其他连接方式怎么接入 |
| 出错处理 | 失败后是重试、回退，还是直接停住 |

## 第一节：先看共同骨架，不要先看表面名字

### 提问

- 为什么这几类产品表面不一样，底层却常常有共同骨架？
- 为什么一上来就看产品名字和宣传语，很容易把差异看偏？
- 为什么这里要先用“共同骨架”这个框架，再去看个体差异？

### 术语

- 中文：共同骨架
- 英文：shared structure

### 一句话解释

这几类产品表面长得不一样，但后续要先抓住它们共同的骨架，再去看差异。

### 你可以先这样理解

- 后续会重点比较：它们是不是都在做“接目标 -> 组织模型上下文（Context）-> 调工具 -> 看结果 -> 再推进”这件事。
- 这样比较的好处，是不会把产品差异误解成“只有模型更强”。
- 先抓共同骨架，才能避免比较时只停留在表面功能列表。

### 你只需要记住

- 先看共同骨架，才能看清差异到底出在哪里。

## 第二节：这一章准备怎么比较 Openclaw、龙虾、Cloud Code

### 提问

- 为什么这一章不打算把三个产品写成三篇分开的产品介绍？
- 为什么统一对比维度，比单独讲故事更有用？
- 为什么这里要把 `Openclaw`、`龙虾`、`Cloud Code` 放进同一组问题里逐项比较？

### 术语

- 中文：对比维度
- 英文：comparison dimensions

### 一句话解释

这一章后续不会只做产品介绍，而是会按同一组维度逐项比较。

### 你可以先这样理解

- 后续会先核实这几个产品公开可见的能力，再按统一维度比较。
- 会尽量避免“这个更强、那个更聪明”这种空话，改成“它的信息从哪来、工具怎么进来、什么时候停”这类可复述的判断。
- 统一维度的好处，是读者以后遇到别的产品也能自己套进去看。

### 表格

| 产品 | 本章后续重点比较什么 |
| --- | --- |
| Openclaw | 它的目标组织方式、工具边界和执行流程 |
| 龙虾 | 它的使用门槛、可复用能力和系统组织方式 |
| Cloud Code | 它的工作场景、连接方式和约束机制 |

### 你只需要记住

- 这章后续不是“讲产品故事”，而是“拆产品结构”。

## 第三节：真正决定体验差异的，往往不是名字，而是这几件事

### 提问

- 为什么很多用户以为差异只在模型，但实际差异常常出在系统层？
- 为什么模型上下文（Context）、工具、权限和终止条件这些变量，会直接决定体验？
- 为什么这里要先列系统变量，再去谈产品差别？

### 术语

- 中文：系统变量
- 英文：system variables

### 一句话解释

同样叫智能体（Agent）产品，体验差异往往来自几个固定变量，而不是一句“它更先进”。

### 你可以先这样理解

- 后续会重点看：模型上下文（Context）够不够清楚、工具接口顺不顺、权限设计严不严、默认流程稳不稳。
- 也会重点看：它更像“开放执行系统”，还是更像“收好边界的任务产品”。
- 这些变量一旦看懂，产品差异就会从“神秘印象”变成“结构判断”。

### 表格

| 关键变量 | 为什么重要 |
| --- | --- |
| 模型上下文（Context）组织 | 决定模型当前到底知道什么 |
| 工具系统 | 决定它能不能真的做事 |
| 技能（Skill）或模板 | 决定常见任务能不能稳定复用 |
| 模型上下文协议（`MCP`）等扩展方式 | 决定外部资源能不能稳定接进来 |
| 权限与确认 | 决定它能不能在真实环境里安全运行 |
| 终止条件 | 决定它会不会一直转下去 |

### 你只需要记住

- 后续真正要比的，是这些结构变量。
- 把这几个变量看懂之后，很多产品差异就没那么神秘了。

## 第四节：作为用户，这一章最后会想帮你建立什么判断

### 提问

- 为什么这一章最后不是想让读者“站队”某个产品？
- 为什么普通用户更需要的是比较方法，而不是一串结论？
- 为什么这里要把判断框架单独拎出来讲？

### 术语

- 中文：用户判断框架
- 英文：user mental model

### 一句话解释

这一章最后想帮读者建立的，不是“站队哪一个产品”，而是“知道该用什么角度看产品”。

### 你可以先这样理解

- 后续会回到两个问题：它到底是靠什么让自己看起来更能干，以及我作为用户到底该怎样用它。
- 也会为下一章埋一个伏笔：一旦出错，我应该优先怀疑哪一层。
- 读者只要带走这套比较方法，后面遇到新的产品也不会完全重新开始。

### 你只需要记住

- 这章最终要给读者的，不是某个结论，而是一套比较方法。
