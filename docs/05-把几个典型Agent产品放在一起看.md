## 📊 先看对比框架

| 对比维度 | 我们重点看什么 |
| --- | --- |
| 🎯 目标入口 | 用户是怎么下达任务的 |
| 📋 上下文（Context）来源 | 它能看到哪些信息，哪些看不到 |
| 🛠️ 工具范围 | 它能调哪些工具，动作边界在哪里 |
| 🔄 执行循环 | 它是一轮一轮推进，还是一次性输出 |
| 🔒 权限与确认 | 哪些动作需要用户同意 |
| 📦 可复用能力 | 技能（Skill）、模板、预设流程怎么组织 |
| 🔌 扩展方式 | 模型上下文协议（Model Context Protocol，`MCP`）或其他连接方式怎么接入 |
| ⚠️ 出错处理 | 失败后是重试、回退，还是直接停住 |

> 💡 **核心思路**：不看宣传语，看结构变量。

## 第一小节：先看共同骨架，不要先看表面名字

### ❓ 提问

- 为什么这几类产品表面不一样，底层却常常有共同骨架？
- 为什么一上来就看产品名字和宣传语，很容易把差异看偏？

### 📝 术语

- **中文**：共同骨架
- **英文**：Shared Structure
- **补充**：不同产品底层共有的核心结构

### 💬 一句话解释

说白了，这几类产品表面长得不一样，但底层都在做同一件事：接目标 → 组织上下文 → 调工具 → 看结果 → 再推进。

### 🧐 你可以先这样理解

市面上有很多智能体（Agent）产品：OpenAI 的 Operator、Claude 的 Computer Use、各种国产的、开源的...

它们的名字不同、界面不同、宣传点不同，但如果你拆开看，底层都是这套骨架：

```
🎯 接收目标 → 📋 组织上下文 → 🧠 模型判断 → 🛠️ 执行工具 → 📥 看结果 → 🔄 循环或结束
```

**为什么先看骨架很重要？**

因为如果不先看骨架，你容易被表面宣传带偏：
- ❌ "这个用了 GPT-4，所以更聪明"
- ❌ "那个是国产的，所以更适合中文"

这些说法都在强调"模型名字"，但体验差异往往来自**上下文怎么组织、工具怎么接、权限怎么设**——这些系统层的设计。

### ✅ 你只需要记住

- 🔍 先看共同骨架，才能看清差异到底出在哪里
- 🎯 骨架就是：目标 → 上下文 → 判断 → 执行 → 结果 → 循环
- 🚫 不要一上来就比"模型名字"，要比"系统设计"

## 第二小节：这一章准备怎么比较 Openclaw、龙虾、Cloud Code

### ❓ 提问

- 为什么这一章不打算把三个产品写成三篇分开的产品介绍？
- 为什么统一对比维度，比单独讲故事更有用？

### 📝 术语

- **中文**：对比维度
- **英文**：Comparison Dimensions
- **补充**：用来比较不同产品的统一视角

### 💬 一句话解释

说白了，这一章不是"讲三个产品故事"，而是按同一组维度逐项比较，让你学会自己判断。

### 🧐 你可以先这样理解

如果每个产品单独介绍，你可能会记住：
- "Openclaw 是 Anthropic 出的"
- "龙虾是国内的"
- "Cloud Code 是云厂商的"

但这些信息对你**选择和使用**帮助不大。

更好的方式是：**用同一组维度，把产品摊开来看**。

| 维度 | Openclaw | 龙虾 | Cloud Code |
| --- | --- | --- | --- |
| 🎯 目标入口 | 自然语言描述任务 | 预设场景 + 自定义 | 代码/配置驱动 |
| 📋 上下文来源 | 当前会话 + 检索 | 知识库 + 会话 | 代码仓库 + 文档 |
| 🛠️ 工具范围 | 浏览器操作为主 | 办公/协作工具 | 云服务/开发工具 |
| 🔒 权限设计 | 高敏感操作需确认 | 按角色分配 | 按 IAM 角色控制 |

这样一比，差异就清楚了：**它们不是"谁更聪明"，而是"为谁设计、解决什么问题"**。

### 📖 本章后续比较重点

| 产品 | 重点看什么 |
| --- | --- |
| 🤖 **Openclaw** | 目标组织方式、工具边界、执行流程 |
| 🦞 **龙虾** | 使用门槛、可复用能力、系统组织方式 |
| ☁️ **Cloud Code** | 工作场景、连接方式、约束机制 |

### ✅ 你只需要记住

- 📊 这章后续不是"讲产品故事"，而是"拆产品结构"
- 🎯 用同一组维度比较，才能看出真正的差异
- 🛠️ 学会这套方法，以后遇到新产品也能自己分析

## 第三小节：真正决定体验差异的，往往不是名字，而是这几件事

### ❓ 提问

- 为什么很多用户以为差异只在模型，但实际差异常常出在系统层？
- 上下文（Context）、工具、权限和终止条件这些变量，为什么能直接决定体验？

### 📝 术语

- **中文**：系统变量
- **英文**：System Variables
- **补充**：决定智能体（Agent）产品体验的关键系统设计因素

### 💬 一句话解释

说白了，同样叫智能体（Agent）产品，体验差异往往来自几个固定的系统变量，而不是一句"它更先进"。

### 🧐 你可以先这样理解

很多人选产品时只看"用的是什么模型"，但真正用起来，这些系统变量才是决定体验的关键：

| 关键变量 | 为什么重要 | 举例 |
| --- | --- | --- |
| 📋 **上下文（Context）组织** | 决定模型当前到底知道什么 | 能不能看到完整的历史记录？ |
| 🛠️ **工具系统** | 决定它能不能真的做事 | 能调哪些 API？能不能操作浏览器？ |
| 📦 **技能（Skill）或模板** | 决定常见任务能不能稳定复用 | 有没有预设的邮件模板？ |
| 🔌 **MCP 等扩展方式** | 决定外部资源能不能稳定接进来 | 能不能接公司的私有系统？ |
| 🔒 **权限与确认** | 决定它能不能在真实环境里安全运行 | 发邮件前要不要我确认？ |
| ⏹️ **终止条件** | 决定它会不会一直转下去 | 什么时候算"做完了"？ |

**这些变量一旦看懂，产品差异就会从"神秘印象"变成"结构判断"**。

### ✅ 你只需要记住

- 🎯 真正要比的，是系统变量，不是模型名字
- 📊 把这几个变量看懂，很多产品差异就没那么神秘了
- 🔍 选产品时，先问自己：它的上下文、工具、权限是怎么设计的？

- **英文**：User Mental Model
- **补充**：帮助用户理解和选择产品的思维方式

### 💬 一句话解释

说白了，这一章最后想帮读者建立的，不是"选哪个产品"，而是"知道该用什么角度看产品"。

### 🧐 你可以先这样理解

## 📝 本章小结

这一章回答了一个实用问题：**同样都叫智能体（Agent），产品之间到底差在哪？**

答案不是"谁更聪明"，而是：

> 🔍 **体验差异主要来自系统层的设计——上下文怎么组织、工具怎么接、权限怎么设、终止条件怎么定。**

| 核心方法 | 一句话总结 |
| --- | --- |
| 🔍 先看骨架 | 目标 → 上下文 → 判断 → 执行 → 结果 → 循环 |
| 📊 再比变量 | 上下文、工具、技能、扩展、权限、终止条件 |
| 🎯 最后判断 | 不是"谁更好"，而是"适合什么场景" |

但掌握了比较方法，还有一个问题：**当我用这些产品时，出了问题该怎么排查？**

是模型不够聪明？还是上下文没给对？还是工具接错了？还是权限卡住了？

这就是下一章要聊的：**普通人怎么真正用好 AI**——从"选产品"到"排问题"的完整指南。
