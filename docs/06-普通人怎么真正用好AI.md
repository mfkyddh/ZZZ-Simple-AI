# 第六章 普通人怎么真正用好 AI

## 读前先想

- 为什么很多人天天在用 AI，还是总觉得它忽好忽坏？
- 真正用好 AI，关键是会提问，还是会判断问题到底卡在哪一层？

## 章节定位

| 维度 | 内容 |
| --- | --- |
| 横向定位 | 这是前面几章的收口章节，会把大模型派（Big Model）和驾驭工程派（Harness Engineering）的理解，一起落到真实使用里。 |
| 本章回扣 | 1. 你平时能不能把 AI 用好，很多时候取决于你有没有把任务和模型上下文（Context）讲清楚。<br>2. 很多故障排查，本质上是在判断问题到底出在模型上下文（Context）、工具、技能（Skill）还是模型上下文协议（Model Context Protocol，`MCP`）这一层。 |

## 先看排查地图

| 常见现象 | 更应该先怀疑什么 |
| --- | --- |
| 回答很空、很泛 | 目标没讲清，或模型上下文（Context）不够 |
| 明明有资料却没用上 | 检索没取到，或资料没进模型上下文（Context） |
| 说自己会做，但没有真的执行 | 工具没接好，或权限没放开 |
| 老是走偏、重复、来回绕 | 执行循环设计不稳，或终止条件不清楚 |
| 同一个任务忽好忽坏 | 技能（Skill）说明不稳，或输入边界不一致 |
| 外部服务完全接不上 | 模型上下文协议（`MCP`）或连接配置有问题 |

## 第一节：先别急着问“它聪不聪明”，先问“任务清不清楚”

### 提问

- 为什么很多使用失败，第一原因不是模型不行，而是任务没说清？
- 为什么普通用户最容易忽略的，其实是任务目标、背景和边界？
- 为什么这里要把“任务表达”放在整个排查顺序的最前面？

### 术语

- 中文：任务表达
- 英文：task framing

### 一句话解释

普通用户能不能把 AI 用好，第一步通常不是换模型，而是把任务目标、背景和边界讲清楚。

### 你可以先这样理解

- 后续会重点讲：什么叫任务太大、目标太散、边界太模糊。
- 也会讲：为什么很多失败，不是模型不会，而是问题本身就没有说清楚。
- 这一步最基础，但也最容易被忽略。

### 示例

- 占位：这里后续会补“帮我搞一下这个项目”为什么太空，以及怎么把它改成更可执行的表达。

### 你只需要记住

- 好用的第一步，不是追新模型，而是先把任务说清楚。

## 第二节：如果结果不对，先查模型上下文（Context）

### 提问

- 为什么很多“答非所问”“回答很空”的问题，第一优先要查模型上下文（Context）？
- 为什么“资料明明在那儿”不等于“模型真的看到了它”？
- 为什么这里要把模型上下文（Context）当成排查的第一大层？

### 术语

- 中文：模型上下文
- 英文：Context

### 一句话解释

很多“答非所问”或“回答很空”的问题，第一优先不是怪模型，而是先查它到底看到了什么。

### 你可以先这样理解

- 后续会重点讲：目标、背景、限制、示例、历史记录，哪些会进入模型上下文（Context）。
- 也会讲：为什么“资料明明在那儿”不等于“模型真的看到了”。
- 这一步本质上是在查输入，而不是在查“智商”。

### 表格

| 要检查的点 | 后续会怎么讲 |
| --- | --- |
| 目标有没有说清楚 | 任务到底要产出什么 |
| 背景有没有补够 | 它知不知道你的真实场景 |
| 限制条件有没有写明 | 它知不知道字数、语气、格式和边界 |
| 参考资料有没有真的进来 | 它看到的是文件名，还是文件内容 |

### 你只需要记住

- 先查模型上下文（Context），是最划算的一步。

## 第三节：如果它说会做，但没真的做到，优先查工具

### 提问

- 为什么很多时候 AI 说“我已经做了”，结果却什么都没发生？
- 为什么“模型会说”不等于“工具真的执行成功”？
- 为什么这里要把工具层单独拎出来查，而不是继续怪模型？

### 术语

- 中文：工具调用
- 英文：Tool Calling

### 一句话解释

很多“嘴上会，手上不会”的问题，本质上是工具层没有真的接通，或者动作边界没放开。

### 你可以先这样理解

- 后续会重点区分：它是不会决定用哪个工具，还是根本没有可用工具。
- 也会重点讲：为什么有时问题不是工具坏了，而是权限、参数或工作目录不对。
- 这一层在真实工作里非常常见，因为“能说”和“能做”本来就是两回事。

### 示例

- 占位：这里后续会补“它说已经改好了文件，但实际没有改动”这类案例。

### 你只需要记住

- 模型表达调用意图，不等于工具真的执行成功。

## 第四节：如果同类任务总是不稳，优先查技能（Skill）

### 提问

- 为什么同一类任务明明重复很多次，结果却还是忽好忽坏？
- 为什么很多不稳定，不是模型突然失常，而是技能（Skill）本身写得不稳？
- 为什么这里要把技能（Skill）单独当成一层来排查？

### 术语

- 中文：技能
- 英文：Skill

### 一句话解释

如果某类任务总是忽好忽坏，常见原因不是模型突然失常，而是技能（Skill）本身的说明、边界和步骤组织得不稳。

### 你可以先这样理解

- 后续会重点讲：什么叫技能（Skill）写得太泛、太散、太长，或者默认假设太多。
- 也会讲：为什么技能（Skill）不是越复杂越好，而是越清楚越好。
- 这一层本质上是在查“可复用做法”有没有被整理好。

### 表格

| 典型症状 | 后续优先检查什么 |
| --- | --- |
| 同一个技能（Skill）每次输出风格都不一样 | 说明是否过宽，限制是否太少 |
| 一进复杂任务就乱 | 步骤是否过多，边界是否不清 |
| 老是跳过关键步骤 | 提示是否把关键动作写得不够明确 |

### 你只需要记住

- 技能（Skill）不是装饰，而是稳定性工具。

## 第五节：如果外部资源接不上，优先查模型上下文协议（`MCP`）

### 提问

- 为什么一旦 AI 需要连接外部资源，问题就不再只是“模型答得好不好”？
- 为什么 `MCP` 出问题时，表面上很像模型太笨，实际却可能是连接根本没通？
- 为什么这里要把模型上下文协议（Model Context Protocol，`MCP`）单独拎成一层？

### 术语

- 中文：模型上下文协议
- 英文：Model Context Protocol
- 简称：`MCP`

### 一句话解释

如果 AI 需要连接外部资源，但总是拿不到、读不对或调不通，就要优先检查模型上下文协议（`MCP`）这一层。

### 你可以先这样理解

- 后续会重点讲：什么是“协议没接上”，什么是“接上了但权限不对”，什么是“能列出来但读不到内容”。
- 也会讲：为什么 `MCP` 出问题时，表面上很像模型太笨，实际上是外部连接根本没通。
- 这一步本质上是在查系统对外部世界的接口。

### 示例

- 占位：这里后续会补“明明挂了知识库，但它还是像没看见一样”这类案例。

### 你只需要记住

- 一旦涉及外部资源，别只盯着模型，要把 `MCP` 这一层也单独看。

## 第六节：这一章最后会给读者一套什么顺序

### 提问

- 为什么普通用户遇到问题时，最需要的不是更多术语，而是一套固定顺序？
- 为什么排查顺序如果不固定，就很容易乱试、重试、越试越乱？
- 为什么这里要把“排查顺序”单独总结出来？

### 术语

- 中文：排查顺序
- 英文：troubleshooting order

### 一句话解释

这一章最后会给读者一套从简单到复杂的排查顺序，而不是把问题都推给“模型不行”。

### 你可以先这样理解

- 后续准备给出一套固定顺序：先看任务，再看模型上下文（Context），再看工具，再看技能（Skill），最后看 `MCP` 和系统连接。
- 这样写的目的，是让普通用户遇到问题时有地方下手，而不是只会反复重试。
- 这也是前面所有章节最后要落到现实使用中的地方。

### 你只需要记住

- 真正用好 AI，不只是会提问，还包括会判断问题出在哪一层。
