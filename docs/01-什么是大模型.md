# 第一章 什么是大模型

## 读前先想

- 大模型明明没有"脑子"，为什么你觉得它好像能"理解"你？
- 为什么学了那么多提示词技巧，最后还是回到"把话说清楚"这个基本功？

## 章节定位

| 维度 | 内容 |
| --- | --- |
| 横向定位 | 这是全项目的共同基础，不讲派系，只讲底层。 |

## 第一小节：本质——按概率生成词元

### 提问

- 为什么大模型有时候答得很准，有时候又答得莫名其妙？
- 如果它真的"懂"我说的话，为什么换种问法，答案就变了？

### 术语

- **中文**：大语言模型
- **英文**：Large Language Model
- **简称**：LLM
- **补充**：词元（token）可以先理解成文字片段

### 一句话解释

说白了，大模型就是个超级会"接话"的人——你起个头，它顺着往下编。

### 你可以先这样理解

你有没有觉得，大模型有时候答得很准，有时候又答得莫名其妙？

其实原因很简单：它只是在根据你给的前文，猜测下一个词最可能是什么。它没有真正的"思考"，也不会提前把整件事想明白再输出。它更像是一个超级会"接话"的伙伴，一直在判断"接下来最可能说什么"。

举个例子：你输入"今天天气真"，模型会看海量文本里"今天天气真"后面最常跟什么——可能是"好"、"热"、"糟糕"——然后按概率选一个续上。一个词接一个词，最后连成一段话。

所以，我们先讲"按概率生成"，不是故意把 AI 讲得冷冰冰，而是让你看到最底层的运作方式。

**为什么要从这个看起来"冷冰冰"的本质讲起？**

因为后面所有的技术——RAG、Agent、工具调用——都建立在这个基础之上。如果你接受"大模型只是在接话"这个事实，后面很多设计选择就会豁然开朗：既然它不会真正"思考"，那我们能做的，就是帮它把"接话的依据"给得更充分。这就是上下文（Context）的重要性。

只有这样，后面再遇到"它为什么答错""它为什么跑偏"时，你才不会觉得神秘。

### 示例

假设你正在赶去开会，眼看要迟到了，你对 AI 说：

> "帮我写一句开会迟到时的道歉话。"

它很可能会回复：

> "不好意思，我刚才路上有点堵，耽误了几分钟。"

这个回复听起来挺像人写的，对吧？但 AI 并不是真的经历过堵车，也不是在"思考"该怎么道歉。它只是在海量的语言材料里发现，"开会迟到"后面跟着"路上有点堵"这样的说法非常常见，所以按概率生成了这个续写。

### 你只需要记住

- LLM 的本质是预测下一个词，不是"思考"或"理解"。
- 它看起来聪明，是因为训练数据里见过太多类似的语言模式。
- 理解这一点，后面很多"奇怪表现"就不奇怪了。

## 第二小节：上下文（Context）——模型能看到的全部信息

### 提问

为什么你给的信息越详细，AI 回得越好？

### 术语

- **中文**：模型上下文
- **英文**：Context
- **补充**：就是模型当前能看到、能参考的全部信息

### 一句话解释

上下文（Context）是模型生成答案时的"依据"——你给得越清楚，它猜得越准。

### 你可以先这样理解

如果第一节回答的是"模型到底在做什么"，这一节回答的就是"模型拿什么来做"。

模型上下文（Context）就是模型当前能看到的全部信息。你的问题、背景、目标、限制、示例，都会成为它预测下一个词时的参考。

说白了，大模型就像一个被蒙住眼睛的人，你告诉它什么，它就只能基于什么来回应。你给的信息笼统，它只能猜个大概；你给的信息具体，它就能有的放矢。

这就是为什么后面各种 AI 技术——RAG、Agent、工具调用——本质上都在做同一件事：**想办法把更多、更准确的信息，塞进模型的上下文里**。

| 信息类型 | 例子 | 会不会进入上下文 | 作用 |
| --- | --- | --- | --- |
| 用户问题 | 帮我写周报 | 会 | 决定任务方向 |
| 背景资料 | 本周做了用户访谈和竞品整理 | 会 | 决定内容贴合度 |
| 限制条件 | 150 字以内，语气正式 | 会 | 决定输出边界 |
| 示例 | 参考上周周报格式 | 会 | 决定表达方式 |

### 示例

如果你只说：

```
帮我写个周报
```

它很可能写得很空，也不一定符合你的场景。

但如果你说：

```
帮我写个周报，面向产品经理，语气简洁，本周做了用户访谈、整理竞品、输出需求文档，控制在 150 字以内
```

它通常就会写得更贴近你的目标。

差别不在于模型突然变聪明了，而在于你给的上下文（Context）更清楚了。

### 你只需要记住

- 上下文（Context）就是模型当前能看到的信息，是它生成答案的唯一依据。
- 你给得越清楚、越完整，结果通常越稳。
- 后面各种 AI 使用方法，核心都和"如何组织上下文"有关。

## 第三小节：为什么上下文这么重要——从本质串联起来看

### 提问

- 既然大模型只是"猜下一个词"，为什么上下文给得多、给得准，答案就能变聪明？
- 这和搜索引擎"找现成答案"有什么区别？

### 一句话解释

上下文就是模型的"线索"。线索越多，它就越知道该调用哪部分"经验"来回答你。

### 你可以先这样理解

把前两小节连起来看：

**第一小节说**，大模型是个概率生成器——它根据前文，猜下一个词最可能是什么。

**第二小节说**，上下文就是它能看到的前文——你给它什么，它就基于什么来猜。

**这一小节想说**：上下文的重要性，正是从这两件事里长出来的。

想象一下这个画面：

> 大模型就像一个读过海量书籍的人，但你每次只能给它看一张纸条。纸条上写什么，它就基于自己读过的书，去猜"这段话接下来最可能说什么"。

- 纸条上只写"写个报告"——它只能猜个大概，因为线索太少。
- 纸条上写"写个产品周报，面向技术团队，本周完成了登录功能优化和接口性能测试"——它就有了足够线索，能调用自己关于"技术周报"的经验，生成更精准的内容。

这就是上下文的威力：**它不是让模型"变聪明"，而是帮模型更好地调用它本来就有的能力。**

模型在训练时已经"读"了海量文本，学会了各种语言模式、知识关联、表达风格。上下文的作用，就是告诉它"现在该调用哪一部分"。

```
训练阶段：模型学会了很多"如果前文是A，后文最可能是B"的模式

使用阶段：你给的前文（上下文）越匹配某个模式，模型就越能准确续写
```

所以，后面你会看到 RAG（检索增强生成）、工具调用、Agent 这些技术——它们本质上都是在想办法**把更丰富、更精准的信息，组织进上下文里**。不是模型突然变强了，而是我们帮它把"依据"给得更充分了。

### 示例

假设你要让 AI 帮你回复一封客户邮件。

**场景一：上下文很单薄**

```
帮我回复这封邮件。
```

AI 根本不知道邮件内容是什么，只能瞎猜一个通用回复，大概率不符合你的需求。

**场景二：上下文充分**

```
客户来信说对我们的定价有疑虑，认为比竞品贵了30%。
客户背景：初创公司，预算敏感，但之前表示很看重我们的售后服务。
请帮我写一封回复，语气专业友好，既要解释价值，又不要显得 defensive。
```

现在 AI 有了客户具体顾虑、背景信息、你的语气要求——它就能调用训练中学到的"商务沟通""异议处理"等模式，生成更精准、更有针对性的回复。

**核心差别**：不是模型突然"理解"了客户关系，而是上下文给了它足够的"线索"，让它能匹配到更合适的语言模式。

### 你只需要记住

- 大模型的能力本质上是"模式匹配"，上下文决定了它匹配什么模式。
- 上下文越充分，模型就越能调用它训练中学到的相关能力。
- 后面各种 AI 技术（RAG、工具调用等），核心都是在解决"如何把更多信息有效塞进上下文"。

## 本章小结

这三小节连在一起，回答了一个最基础的问题：**大模型到底是什么？**

- 它不是"思考"，是按概率生成词元。
- 它生成得好不好，取决于上下文给得够不够。
- 上下文是模型能力的"开关"——给对了，它就能调用训练中的相关知识；给少了，它就只能瞎猜。

理解这一点，你就抓住了后面所有技术的起点。不管是 RAG 去检索资料，还是 Agent 去调用工具，本质上都是在说同一句话：**帮模型把上下文凑齐**。

明白了大模型只是"接话"和"看纸条"，下一个问题自然就来了：既然它只会接话，那它是怎么从"聊天框"一步步进化到"能做事"的？这就是下一章要讲的故事。
