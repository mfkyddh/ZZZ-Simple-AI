## 📖 第一小节：为什么AI编程总是"改着改着就乱了"

### ❓ 提问

- 为什么让AI改一段代码，它经常改着改着就偏离目标了？
- 同样的需求，为什么AI有时做得很好，有时却"放飞自我"？

### 📝 术语

- **Plan模式（Plan Mode）**：先制定详细计划，经确认后再执行的工作模式
- **执行模式（Act Mode）**：边想边干、直接动手改代码的模式

### 💬 一句话解释

AI编程需要"先规划再动手"，因为模型本身不会自动"三思而后行"——它只会根据当前看到的内容，预测下一个最可能出现的词。

### 🧐 你可以先这样理解

想象你请了一个装修工人（AI）来改造你的房子：

**执行模式（Act Mode）** 就像工人拿到钥匙就进场，边看边改：
- 看到厨房不顺眼，拆了重装
- 拆完发现水管位置不对，又改水管
- 改完水管发现影响客厅布局，又把客厅拆了
- 三天后你发现房子一团糟，原始格局也找不回来了

**Plan模式** 就像工人先花一天时间：
- 仔细测量每个房间
- 画出设计图纸
- 列出材料清单和施工顺序
- 给你看过方案、确认无误后再动手

AI模型本质是按概率"接话"的。任务越短，它越不容易跑偏；任务越长，它越容易"走一步看一步"，逐渐被新信息带偏，忘记最初的目标。

这就是为什么长任务容易"改着改着就乱了"——**上下文被新信息淹没了**。

### 📚 示例

**场景**：让AI给现有项目添加用户登录功能

**执行模式下的AI**：
1. 先看到需要登录页面，就创建一个 login.html
2. 然后发现需要后端接口，就创建一个 login.php
3. 写着写着发现需要数据库，就去改数据库配置
4. 改数据库时发现权限系统不完善，又去改权限系统
5. 最后登录功能还没做好，其他部分却被改得乱七八糟

**Plan模式下的AI**：
1. **研究阶段**：通读整个代码库，理解现有架构
2. **规划阶段**：生成实施计划——需要改哪些文件、按什么顺序、每步做什么
3. **确认阶段**：把计划给你看，你确认无误
4. **执行阶段**：严格按照计划一步步实施

### ✅ 你只需要记住

1. AI模型是按概率"接话"的，**任务越长越容易跑偏**
2. Plan模式的核心是**"先规划再动手"**，不是让AI变聪明，而是强制它"三思而后行"
3. 这属于**主线二（驾驭工程派）**——通过设计"作业环境"让模型可靠工作

## 📖 第二小节：Plan模式——让AI"先想后做"

### ❓ 提问

- Plan模式具体是怎么工作的？
- 主流AI编程工具都支持吗？

### 📝 术语

- **Plan模式（Plan Mode）**：创建只读研究阶段，在做任何代码更改之前先分析和规划
- **只读研究阶段**：AI只能查看代码，不能修改，用于充分理解代码库

### 💬 一句话解释

Plan模式给AI配了一个"项目经理"角色，强制它先进入"只读研究"状态理解代码库，生成实施计划，经你批准后再动手。

### 🧐 你可以先这样理解

回到装修工人的比喻：

**执行模式** = 工人拿到钥匙就进场拆墙
**Plan模式** = 工人先花一天"量房"——只看不动手

Plan模式的设计有三条"克制"：

1. **不改代码，直到你点"开始实施"**
   - AI在这个阶段只能"看"和"想"，不能"动手"
   - 你掌握最终决策权

2. **只读探索，充分理解上下文**
   - AI会通读相关文件，理解项目结构
   - 这就像给模型**扩展上下文**——让它看到更多相关信息

3. **生成可审计的计划文档**
   - 不是黑盒操作，而是产出清晰的计划清单
   - 你可以审查、修改、确认每一步

**为什么这能解决问题？**

- **回扣本质一（概率模型）**：把"规划"和"执行"分离，降低执行阶段的随机性
- **回扣本质二（上下文）**：规划阶段在**扩展上下文**（充分理解代码库），执行阶段在**约束上下文**（按计划走，不被新信息带偏）

### 📚 示例

**主流AI编程工具的Plan模式对比**：

| 产品 | 模式名称 | 核心特点 |
|:---|:---|:---|
| **Claude Code** | Plan Mode | 最完整的实现，支持复杂任务规划，生成详细实施计划 |
| **GitHub Copilot** | Plan模式 | 集成在IDE里，适合日常开发任务 |
| **Cursor** | Agent/Ask模式 | 通过对话方式自然切换规划和执行 |

**使用Claude Code Plan Mode的典型流程**：

```
你：我想给这个项目添加用户权限系统

Claude：我来启动Plan模式分析这个项目...
[只读研究阶段：通读代码库，理解现有架构]

Claude：我分析了项目结构，这是我的实施计划：
1. 创建用户表迁移文件
2. 添加认证中间件
3. 修改路由配置
4. 创建登录/注册页面
5. 添加权限检查逻辑

你：第3步我有点担心，会不会影响现有API？

Claude：好问题。让我重新检查一下API路由...
[调整计划]

你：确认执行

Claude：[开始按步骤实施]
```

### ✅ 你只需要记住

1. Plan模式的核心是**"规划与执行分离"**——不是让AI变聪明，而是给它配一个"项目经理"
2. 规划阶段**扩展上下文**（理解全貌），执行阶段**约束上下文**（按计划走）
3. 适合**复杂、多步骤、影响范围广**的任务

## 📖 第三小节：从"写代码"到"写规格"——SDD规范驱动

### ❓ 提问

- Plan模式解决了单次任务的规划问题，但项目级的代码质量如何保证？
- 如果AI生成了代码，半年后出了问题，谁来负责？

### 📝 术语

- **SDD（Spec-Driven Development）**：规范驱动开发，以规格说明书为核心驱动代码生成
- **规格即代码（Spec as Code）**：规格是首要产物，代码从规格派生

### 💬 一句话解释

SDD的核心是"规格即代码"——人写规格，AI生成代码，每行代码都能追溯到规格条款，让不确定性被关在笼子里。

### 🧐 你可以先这样理解

想象你是一家餐厅的老板：

**传统开发** = 你直接教厨师（AI）做菜
- 你说："做一份红烧肉"
- 厨师凭经验做，可能每次味道都不一样
- 半年后你问："为什么今天这么咸？"——没人记得当时的配方

**SDD** = 你先写菜谱（规格），厨师按菜谱做
- 你写："红烧肉规格——五花肉500g、生抽2勺、老抽1勺、冰糖30g..."
- 厨师按菜谱做，出品稳定
- 半年后味道变了，你查菜谱就知道哪里出了问题

**SDD的三大核心原则**：

1. **单一事实来源（SSOT）**
   - 规格文档是项目唯一合法的需求定义
   - 代码必须与规格严格对齐

2. **约束驱动**
   - AI在规格设定的边界内创作，而非自由发挥
   - 这就是**回扣本质一**——用规格约束概率生成

3. **可回溯性**
   - 每行AI生成的代码都能追溯到规格中的具体条款
   - 规格是最结构化的**上下文组织形式**（**回扣本质二**）

**为什么这很重要？**

AI生成代码很快，但"生成"不等于"正确"，更不等于"可维护"。当项目越来越大，你需要知道：
- 这段代码为什么这样写？
- 需求变了，该改哪些地方？
- 新成员如何快速理解项目？

规格文档回答了这些问题。在SDD里，**人写规格，AI生成代码，代码只是规格的副产品**。

### 📚 示例

**对比"直接生成"和"规格驱动"**：

**方式一：直接让AI写功能**
```
你：给我写一个用户登录功能
AI：[生成代码]
半年后：这段代码谁写的？为什么这样设计？不知道。
```

**方式二：SDD规范驱动**
```
1. 人写规格文档（spec.md）：
   - 功能：用户登录
   - 输入：用户名、密码
   - 验证规则：用户名长度3-20字符，密码必须包含字母和数字...
   - 错误处理：用户名不存在→提示"用户名或密码错误"...

2. AI根据规格生成代码

3. 验证：代码是否符合规格？

4. 半年后：查spec.md就知道当初的设计逻辑
```

**代表项目**：

| 项目 | 核心思想 | 特点 |
|:---|:---|:---|
| **OpenSpec** | 规格文件驱动AI生成代码 | 用形式化规格约束AI输出 |
| **Superpowers** | 可复用的Skills | Skills本质上就是**规范模板**——把常见任务的规格封装起来 |

### ✅ 你只需要记住

1. SDD的核心是**"规格即代码"**——人写规格，AI生成代码
2. 规格是**单一事实来源**，每行代码都能追溯到规格条款
3. 这是**驾驭工程派**的终极形态——用规范把概率模型的不确定性关进笼子里

---

### 📊 核心脉络回顾

| 层次 | 问题 | 解决方案 | 回扣本质 |
|:---|:---|:---|:---|
| 单次任务 | AI改代码容易乱 | Plan模式（先规划再执行） | 规划与执行分离，降低随机性 |
| 项目规范 | AI代码难维护 | SDD（规格即代码） | 用规格约束概率生成 |
| 整体思路 | 模型能力≠工程可靠性 | 驾驭工程派——设计"作业环境" | 不关心模型多强，关心环境设计 |

---

### 🔗 回到两个本质

| 本质 | 在本章的体现 |
|:---|:---|
| 🎯 **本质一（概率模型）** | Plan模式通过"规划与执行分离"降低执行阶段的随机性；SDD通过"规格约束"把概率生成限制在可控范围内 |
| 📋 **本质二（上下文）** | Plan模式的规划阶段在**扩展上下文**（研究代码库），SDD的规格文档是最结构化的**上下文组织形式** |
