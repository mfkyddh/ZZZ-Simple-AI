# 🧠 第三章 大模型派与驾驭工程派：一体两面的双螺旋

## 🤔 读前先想

- 第二章里，RAG 是去查资料、工具调用是让 AI 能操作东西、工作流是固定步骤——这些好像都是在"帮"模型。那模型自己到底能独立完成多少？
- 为什么同样是做 AI，有的公司拼命升级模型本身，有的公司拼命搭建 RAG、工具、工作流这些外围系统？

## 🧬 先看两种思路——一体两面的双螺旋

| 思路 | 它更看重什么 | 你可以先怎么理解 |
| --- | --- | --- |
| 🧠 **大模型派（Big Model）** | 模型本身是不是越来越强 | 更相信"脑子"会越来越强 |
| 🛠️ **驾驭工程派（Harness Engineering）** | 模型外部的系统是不是足够稳 | 更相信"系统"要先把模型的输出稳稳地承接住 |

> 💡 **一句话总结**：AI 要真正解决问题，既需要模型能力进化，也需要外部系统把这股能力组织起来——**两者不是对立，而是像 DNA 双螺旋一样，并行发展、相互缠绕、缺一不可**。

---

## 📖 第一小节：为什么会有这两种做法

### ❓ 提问

- 为什么同样做 AI，最后会分出两种关注点？
- 为什么只谈模型或者只谈系统，都不够解释真实产品？

### 📝 术语

- **中文**：两条主线
- **英文**：Two Approaches
- **补充**：指大模型派（Big Model）和驾驭工程派（Harness Engineering）两种不同的侧重点

### 💬 一句话解释

> 💡 **说白了，做 AI 产品时会遇到两类不同的问题：一类靠模型变强解决，一类靠系统设计解决，所以自然分出了两条思路。**

### 🧐 你可以先这样理解

想象你在搭一辆赛车：

- 🏎️ **大模型派**更关心：发动机能不能更强劲？马力能不能更大？
- 🔧 **驾驭工程派**更关心：结构是否稳定？制动是否灵敏？抓地是否牢固？

**这不是二选一，而是必须同时做好。** 光有好发动机，车会散架；光有好车架，跑不快。

更重要的是，这两者是**相互促进**的：
- 发动机越强，对车架的要求越高，推动车架技术进化
- 车架越稳，越能承受更强发动机的输出，让发动机有用武之地

所以这不是纯理论争论，也不是路线之争，而是做产品时必然会遇到的两类互补问题。

### 📖 示例

面对同一个需求，两类人会先问不同的问题：

**大模型派先问**：
> "模型能不能自己把复杂任务一次性想明白？"

**驾驭工程派先问**：
> "就算模型会想，知识库、工具、权限和审计谁来接？"

这两个问题的出发点不同，就会自然走向两种不同的侧重。

### ✅ 你只需要记住

- 🎭 这不是纯理论争论，而是做产品时自然出现的两种思路
- 🧠 一种更强调模型进化，一种更强调系统组织
- 🎯 两条路不是对立的，而是缺一不可

---

### 📖 具体案例：看产品怎么站队

理论说得再多，不如看真实产品怎么选择。

| 产品/技术 | 更偏向哪派 | 怎么看出来的 |
|:---|:---|:---|
| **DeepSeek-R1 / Kimi k2.5** | 🧠 大模型派 | 强调模型自己推理、规划，减少外部干预 |
| **扣子 Coze / Dify** | 🛠️ 驾驭工程派 | 提供大量外部工具、流程框架、RAG组件 |
| **ChatGPT / Claude** | 两者融合 | 既有强模型，又有完善的系统层设计 |

**洞察**：没有产品"纯粹"属于某一派，只是**侧重点不同**。真正成熟的产品，往往是两派的融合。

## 📖 第二小节：大模型派——相信脑子够强，事情自然成

### ❓ 提问

- 为什么随着模型越来越强，会有人想把更多事情直接交给模型本身？
- 为什么这一派会觉得很多外部补丁以后可能变轻？

### 📝 术语

- **中文**：大模型派
- **英文**：Big Model
- **补充**：更强调模型本身持续变强的思路

### 💬 一句话解释

> 💡 **说白了，大模型派相信：只要"脑子"够强，很多原来需要外部补的能力，以后可以交给模型自己搞定。**

### 🧐 你可以先这样理解

如果模型的理解能力、推理能力和长上下文处理能力不断增强，会发生什么？

- ✅ 一些原来很依赖外部技巧的做法，可能会变轻，甚至被模型吸收
- ✅ 一些复杂的提示词工程，可能变得没那么必要
- ✅ 一些原来必须拆成多步骤的任务，模型可能一次性就能做好

这条思路更关心的是：**模型本身能不能做得更多、更稳、更聪明**。

它的基本判断是，很多复杂度最终可以往模型内部收。

### 📖 示例

更强的模型可能带来这些变化：

| 原来需要外部补的 | 模型变强后可能的变化 |
| --- | --- |
| 很细碎的提示词技巧 | 模型更能直接理解自然语言意图 |
| 复杂的多步骤拆解 | 模型更能自己规划步骤 |
| 大量的示例（Few-shot） | 模型更能从简单描述里举一反三 |
| 固定的工具调用格式 | 模型更能灵活决定何时用、怎么用 |

---

### 📖 行业发展：大模型派的实际突破

技术发展从来不是单方向的。当我们关注"驾驭工程派"如何搭建外部系统时，**大模型本身也在快速进化**，不断把原来需要外部技巧的能力"收"进模型内部。

**案例 1：思考链（Chain-of-Thought, CoT）的演进**

| 阶段 | 做法 | 变化 |
| :--- | :--- | :--- |
| 早期 | 人必须在提示词里写"让我们一步步思考" | 靠外部技巧引导 |
| GPT-4 时代 | 模型开始能自动展示推理过程 | 模型自己学会"慢思考" |
| DeepSeek-R1 / Kimi k2.5 | 模型内置强大的推理能力，自动拆解复杂问题 | **推理成为模型原生能力** |

**关键洞察**：从"人教模型怎么思考"到"模型自己学会思考"——**复杂度从外部技巧转移到了模型内部**。

**案例 2：工具调用能力的增强**

| 阶段 | 做法 | 变化 |
| :--- | :--- | :--- |
| 早期 | 必须用固定格式（如 JSON）告诉模型何时调用工具 | 严格的外部协议 |
| GPT-4 / Claude 3 | 模型能更自然地理解工具用途，调用更灵活 | 理解能力增强 |
| 最新模型 | 模型能自主决定调用时机、参数调整，甚至错误修正 | **工具使用越来越像本能** |

**关键洞察**：模型不再需要"被精确指挥"才能使用工具，而是**越来越能自主判断**。

**案例 3：长上下文处理的飞跃**

| 阶段 | 上下文长度 | 能力边界 |
| :--- | :--- | :--- |
| GPT-3 时代 | 2K-4K tokens | 只能处理短文 |
| GPT-4 / Claude 2 | 32K-100K tokens | 能读长文档 |
| GPT-4o / Claude 3 | 128K-200K tokens | 能读整本书、大量代码库 |
| Gemini 1.5 Pro / Kimi K2.5 | **1M+ tokens（一兆）** | 能读整本长篇小说、海量代码库、长视频内容 |

**关键洞察**：上下文窗口的扩展，让模型能**一次性处理更复杂的任务**，减少了对"分段处理"等外部技巧的依赖。

> 💡 **案例 4和5（多模态能力、代码能力）**：模型还在持续进化，从"能读文字"到"能看懂听懂"，从"能写代码"到"理解软件工程"——核心趋势是**专业能力不断内化为模型能力**。

---

### 🔄 双向演进：行业真相

> **大模型派和驾驭工程派不是谁在取代谁，而是双线并行、相互刺激。**

- 模型变强了 → 对系统的要求更高（需要承接更强的能力）
- 系统做稳了 → 能释放模型更多潜力（让模型有用武之地）

这就是"双螺旋"的真实写照。

### ✅ 你只需要记住

- 🧠 大模型派更看重模型能力本身的进化
- 📈 它相信"脑子变强"会带来整体能力提升
- 🎯 很多复杂度最终可以往模型内部收

---

💭 **刚才我们看了大模型派的主张**——相信模型自己会越变越强。但就算模型再聪明，也有很多事是它做不到的——比如真的去发邮件、查日历，或者做权限控制。这就引出了另一种思路：与其等模型变强，不如先把系统搭好。

## 📖 第三小节：驾驭工程派——模型再聪明，也得有系统托底

### ❓ 提问

- 为什么就算模型继续变强，很多团队还是会坚持先把系统做稳？
- 为什么真实工作里，模型本身不等于知识、工具、权限和流程？

### 📝 术语

- **中文**：驾驭工程派
- **英文**：Harness Engineering
- **补充**：更强调模型外部系统组织能力的思路

### 💬 一句话解释

> 💡 **说白了，驾驭工程派相信：模型能力要真正落地，关键还在于模型外部的系统设计。**

### 🧐 你可以先这样理解

模型再强，也有它做不到的事：

- ❌ 模型本身不会自己接企业知识库
- ❌ 模型本身不会真正执行工具（发邮件、查日历、操作数据库）
- ❌ 模型本身不等于权限控制、验证机制、审计流程

所以要让 AI 在真实工作里稳定做事，还需要**外部系统**把信息、工具、流程和约束组织起来。

这条思路的核心是：**先让系统稳稳地承接住模型的能力，再妥善地组织好、用起来**。

### 📖 示例

这些都需要系统层的设计：

| 系统层能力 | 为什么重要 |
| --- | --- |
| 🔌 知识库接入 | 把公司文档、实时信息接到模型上下文里 |
| 🛠️ 工具执行 | 真正去发邮件、查日历、操作数据库 |
| 🔒 权限控制 | 哪些操作需要审批，哪些可以自动执行 |
| 📝 审计日志 | 模型做了什么，要能追溯、能复盘 |
| ⚠️ 风险确认 | 高风险动作前停下来等人确认 |

### ✅ 你只需要记住

- 🛠️ 这条思路更看重系统设计
- 🎯 它相信"系统做稳"是 AI 真正落地的关键
- 🔒 模型本身不等于权限、验证、审计和执行层

---

💭 **刚才我们分别看了两派的核心主张**：大模型派相信模型自己会越变越强，驾驭工程派相信系统才能把能力稳稳接住。现在问题来了——前两章学过的那些技术，到底哪些更偏模型、哪些更偏系统？

### 🧐 把技术放回两条路里看

| 技术 | 更偏向哪条路 | 为什么 |
| :--- | :---: | :--- |
| 🔍 检索增强生成（RAG） | 🛠️ 驾驭工程派 | 它通过外部检索给模型补资料 |
| 🛠️ 工具调用（Tool Calling） | 🛠️ 驾驭工程派 | 真正执行工具的是外层应用，不是模型本身 |
| 📋 工作流（Workflow） | 🛠️ 驾驭工程派 | 它先用外部流程把步骤固定下来，确保稳定性 |
| 🤖 智能体（Agent） | 🧠🛠️ 两边都重要 | 一边依赖更强模型，一边依赖更完整的系统组织 |
| 📦 技能（Skill） | 🛠️ 驾驭工程派 | 它把常用做法提前整理成可复用的任务包 |

**这样看的好处**：同一个技术名字，背后往往更偏向某一类问题。放回两条主线里，整套知识会更成体系。

## 📝 本章要义

> 💡 **一句话核心结论**：大模型派与驾驭工程派不是对立的两条路线，而是**一体两面的双螺旋**——模型能力进化需要系统能力承接，系统能力发展又释放模型潜力，两者相辅相成、并行发展。

### 🔗 核心脉络回顾：为什么会有两条路

| 小节 | 看到的"现象" | 抓住的"本质" | 递进关系 |
| :--- | :--- | :--- | :--- |
| 📖 第1节 | 做 AI 产品时关注点不同 | **两类互补问题需要两种解决思路** | 确立分析框架 |
| 📖 第2节 | 有人拼命升级模型 | **相信"脑子"够强，能解决更多问题** | 解析大模型派 |
| 📖 第3节 | 有人拼命搭建系统 | **相信"系统"够稳，才能承接能力** | 解析驾驭工程派 |

**技术归类速查**：

| 技术 | 归属 | 核心逻辑 |
| :--- | :---: | :--- |
| RAG、工具调用、工作流、Skill | 🛠️ 驾驭工程派 | 通过外部系统补能力 |
| Agent | 🧠🛠️ 两者融合 | 既需强模型，也需好系统 |

### 🎯 回到两个本质：从第一性原理出发

**本质一：概率模型——两派的共同基础**

| 问题 | 深入理解 |
| :--- | :--- |
| 两派的出发点是什么？ | 都承认模型是概率生成系统，有不确定性 |
| 两派的分歧在哪？ | **对"如何克服不确定性"的回答不同** |
| 大模型派的回答 | 让模型本身变强，减少不确定性 |
| 驾驭工程派的回答 | 用系统承接不确定性，控制风险 |

**本质二：上下文——两派的共同战场**

| 问题 | 深入理解 |
| :--- | :--- |
| 两派都在解决什么问题？ | **如何把信息有效组织进上下文** |
| 大模型派怎么做？ | 让模型自己处理更长、更复杂的上下文 |
| 驾驭工程派怎么做？ | 用外部系统补充、整理、管理上下文 |
| 核心规律 | **不管哪条路，最终都回到上下文组织** |

**两个本质的关系**：

```
本质一（概率模型）：承认不确定性是内在属性
         ↓
    引出两种应对策略
         ↓
本质二（上下文组织）：
         • 大模型派：模型自己处理更多上下文
         • 驾驭工程派：系统帮忙组织上下文
         • 实际产品：两者融合，相辅相成
```

---

### 🧩 核心洞察：模型进化 vs 系统承接

AI 产品设计的核心问题始终是——

> **模型能力在进化，但永远有不确定性；系统需要承接这种能力，同时控制风险。**

这不是"二选一"的问题，而是**必须同时做好**的问题：
- 只有模型没有系统 → 能力无法落地，风险不可控
- 只有系统没有模型 → 缺乏智能内核，变成传统软件

**双螺旋的本质**：
- 模型越强 → 对系统要求越高 → 推动系统进化
- 系统越稳 → 越能释放模型潜力 → 让模型有用武之地

---

### 🔭 趋势展望：什么会消失，什么会留下

模型变强了，是不是很多外部设计就没必要了？

**不是所有外部能力都一样**，可以分为两类：

| 类型 | 特征 | 未来趋势 |
| :--- | :--- | :--- |
| 🩹 **临时补丁** | 只是因为旧模型不够强才存在 | 会逐渐简化 |
| 🏗️ **系统层能力** | 无论模型多强都必须保留 | 长期存在 |

**临时补丁的例子**：
- 过细的提示词技巧
- 僵化的步骤拆分
- 大量的示例样本

**系统层能力的例子**：
- 工具执行层（真正操作外部系统）
- 权限控制、验证机制、审计日志
- 业务规则和约束条件

**关键洞察**：

> 模型变强后，**为弥补旧模型短板而存在的设计会简化**，但**负责组织模型能力、确保稳定落地的系统层能力不会消失**。

---

### 📖 自己动手：分析你用的产品

**实验任务**：判断你常用的 AI 产品更偏向哪一派

**步骤**：
1. 选一个你常用的 AI 产品
2. 回答以下问题：
   - **产品做什么？** （生成内容？执行任务？辅助决策？）
   - **模型做什么？** （直接生成结果？提供建议？规划步骤？）
   - **系统做什么？** （检索资料？调用工具？控制流程？验证结果？）
   - **如果模型突然失灵，产品还能用吗？** （能 → 偏工程派；不能 → 偏模型派）
3. 判断它更偏向哪一派，并说明理由

**案例分析框架**：

| 维度 | 大模型派特征 | 驾驭工程派特征 |
| :--- | :--- | :--- |
| 产品核心 | 模型输出就是产品 | 模型只是组件之一 |
| 失败表现 | 模型错了，产品就错了 | 模型错了，系统能兜底 |
| 技术重点 | 提示工程、微调 | 工具集成、流程设计、权限控制 |

**思考问题**：
- 这个产品为什么选择这种侧重？
- 它的场景更适合哪种策略？
- 如果让你改进，你会加强哪一派的能力？

### 🌉 为下一章铺垫

明白了大模型派和驾驭工程派是相辅相成的双螺旋，下一个问题自然来了：

**智能体（Agent）这个产品形态，是怎么把这两条主线融合在一起的？**

它既需要强大的模型能力（大模型派），也需要完善的系统设计（驾驭工程派）。一个真正好用的智能体，必须同时做好两者——它是两条主线交汇的**结晶点**。

这就是下一章要聊的：**智能体是怎么"干活"的**——我们会看到，八部件框架如何同时体现两派的思想。
