# 🧠 第三章 大模型派与驾驭工程派：一体两面的双螺旋

## 🤔 读前先想

> 💭 在往下看之前，不妨先想想：

- 第二章里，RAG 是去查资料、工具调用是让 AI 能操作东西、工作流是固定步骤——这些好像都是在"帮"模型。那模型自己到底能独立完成多少？
- 为什么同样是做 AI，有的公司拼命升级模型本身，有的公司拼命搭建 RAG、工具、工作流这些外围系统？

## 🧬 先看两种思路——一体两面的双螺旋

| 思路 | 它更看重什么 | 你可以先怎么理解 |
| --- | --- | --- |
| 🧠 **大模型派（Big Model）** | 模型本身是不是越来越强 | 更相信"脑子"会越来越强 |
| 🛠️ **驾驭工程派（Harness Engineering）** | 模型外部的系统是不是足够稳 | 更相信"系统"要先把模型的输出稳稳地承接住 |

> 💡 **一句话总结**：AI 要真正解决问题，既需要模型能力进化，也需要外部系统把这股能力组织起来——**两者不是对立，而是像 DNA 双螺旋一样，并行发展、相互缠绕、缺一不可**。

---

## 第一小节：为什么会有这两种做法

### ❓ 提问

- 为什么同样做 AI，最后会分出两种关注点？
- 为什么只谈模型或者只谈系统，都不够解释真实产品？

### 📝 术语

- **中文**：两条主线
- **英文**：Two Approaches
- **补充**：指大模型派（Big Model）和驾驭工程派（Harness Engineering）两种不同的侧重点

### 💬 一句话解释

说白了，做 AI 产品时会遇到两类不同的问题：一类靠模型变强解决，一类靠系统设计解决，所以自然分出了两条思路。

### 🧐 你可以先这样理解

想象你在搭一辆赛车：

- 🏎️ **大模型派**更关心：发动机能不能更强劲？马力能不能更大？
- 🔧 **驾驭工程派**更关心：车架稳不稳？刹车灵不灵？轮胎抓不抓地？

**这不是二选一，而是必须同时做好。** 光有好发动机，车会散架；光有好车架，跑不快。

更重要的是，这两者是**相互促进**的：
- 发动机越强，对车架的要求越高，推动车架技术进化
- 车架越稳，越能承受更强发动机的输出，让发动机有用武之地

所以这不是纯理论争论，也不是路线之争，而是做产品时必然会遇到的两类互补问题。

### 📖 示例

面对同一个需求，两类人会先问不同的问题：

**大模型派先问**：
> "模型能不能自己把复杂任务一次性想明白？"

**驾驭工程派先问**：
> "就算模型会想，知识库、工具、权限和审计谁来接？"

这两个问题的出发点不同，就会自然走向两种不同的侧重。

### ✅ 你只需要记住

- 🎭 这不是纯理论争论，而是做产品时自然出现的两种思路
- 🧠 一种更强调模型进化，一种更强调系统组织
- 🎯 两条路不是对立的，而是缺一不可

---

## 📖 第二小节：大模型派——相信"脑子"够强，事情自然成

### ❓ 提问

- 为什么随着模型越来越强，会有人想把更多事情直接交给模型本身？
- 为什么这一派会觉得很多外部补丁以后可能变轻？

### 📝 术语

- **中文**：大模型派
- **英文**：Big Model
- **补充**：更强调模型本身持续变强的思路

### 💬 一句话解释

说白了，大模型派相信：只要"脑子"够强，很多原来需要外部补的能力，以后可以交给模型自己搞定。

### 🧐 你可以先这样理解

如果模型的理解能力、推理能力和长上下文处理能力不断增强，会发生什么？

- ✅ 一些原来很依赖外部技巧的做法，可能会变轻，甚至被模型吸收
- ✅ 一些复杂的提示词工程，可能变得没那么必要
- ✅ 一些原来必须拆成多步骤的任务，模型可能一次性就能做好

这条思路更关心的是：**模型本身能不能做得更多、更稳、更聪明**。

它的基本判断是，很多复杂度最终可以往模型内部收。

### 📖 示例

更强的模型可能带来这些变化：

| 原来需要外部补的 | 模型变强后可能的变化 |
| --- | --- |
| 很细碎的提示词技巧 | 模型更能直接理解自然语言意图 |
| 复杂的多步骤拆解 | 模型更能自己规划步骤 |
| 大量的示例（Few-shot） | 模型更能从简单描述里举一反三 |
| 固定的工具调用格式 | 模型更能灵活决定何时用、怎么用 |

### ✅ 你只需要记住

- 🧠 大模型派更看重模型能力本身的进化
- 📈 它相信"脑子变强"会带来整体能力提升
- 🎯 很多复杂度最终可以往模型内部收

---

💭 **刚才我们看了大模型派的主张**——相信模型自己会越变越强。但就算模型再聪明，也有很多事是它做不到的——比如真的去发邮件、查日历，或者做权限控制。这就引出了另一种思路：与其等模型变强，不如先把系统搭好。

## 📖 第三小节：驾驭工程派——模型再聪明，也得有系统托底

### ❓ 提问

- 为什么就算模型继续变强，很多团队还是会坚持先把系统做稳？
- 为什么真实工作里，模型本身不等于知识、工具、权限和流程？

### 📝 术语

- **中文**：驾驭工程派
- **英文**：Harness Engineering
- **补充**：更强调模型外部系统组织能力的思路

### 💬 一句话解释

说白了，驾驭工程派相信：模型能力要真正落地，关键还在于模型外部的系统设计。

### 🧐 你可以先这样理解

模型再强，也有它做不到的事：

- ❌ 模型本身不会自己接企业知识库
- ❌ 模型本身不会真正执行工具（发邮件、查日历、操作数据库）
- ❌ 模型本身不等于权限控制、验证机制、审计流程

所以要让 AI 在真实工作里稳定做事，还需要**外部系统**把信息、工具、流程和约束组织起来。

这条思路的核心是：**先让系统稳稳地承接住模型的能力，再妥善地组织好、用起来**。

### 📖 示例

这些都需要系统层的设计：

| 系统层能力 | 为什么重要 |
| --- | --- |
| 🔌 知识库接入 | 把公司文档、实时信息接到模型上下文里 |
| 🛠️ 工具执行 | 真正去发邮件、查日历、操作数据库 |
| 🔒 权限控制 | 哪些操作需要审批，哪些可以自动执行 |
| 📝 审计日志 | 模型做了什么，要能追溯、能复盘 |
| ⚠️ 风险确认 | 高风险动作前停下来等人确认 |

### ✅ 你只需要记住

- 🛠️ 这条思路更看重系统设计
- 🎯 它相信"系统做稳"是 AI 真正落地的关键
- 🔒 模型本身不等于权限、验证、审计和执行层

---

💭 **刚才我们分别看了两派的核心主张**：大模型派相信模型自己会越变越强，驾驭工程派相信系统才能把能力稳稳接住。现在问题来了——前两章学过的那些技术，到底哪些更偏模型、哪些更偏系统？

## 📖 第四小节：技术归类——放回两条路里看

> 💡 **一句话理解**：看一个技术，先问它在补"模型"还是在补"系统"。

### 📊 技术对照表

| 技术 | 更偏向哪条路 | 为什么 |
| :--- | :---: | :--- |
| 🔍 检索增强生成（RAG） | 🛠️ 驾驭工程派 | 它通过外部检索给模型补资料 |
| 🛠️ 工具调用（Tool Calling） | 🛠️ 驾驭工程派 | 真正执行工具的是外层应用，不是模型本身 |
| 📋 工作流（Workflow） | 🛠️ 驾驭工程派 | 它先用外部流程把步骤固定下来，确保稳定性 |
| 🤖 智能体（Agent） | 🧠🛠️ 两边都重要 | 一边依赖更强模型，一边依赖更完整的系统组织 |
| 📦 技能（Skill） | 🛠️ 驾驭工程派 | 它把常用做法提前整理成可复用的任务包 |

这样看的好处：同一个技术名字，背后往往更偏向某一类问题。放回两条主线里，整套知识会更成体系，而不是彼此孤立的小知识点。

## 第五小节：以后哪些会变轻，哪些会留下来

### ❓ 提问

- 为什么不是所有外部系统能力都会被更强的模型直接吃掉？
- 模型变强了，是不是很多外部设计就没必要了？

### 📝 术语

- **中文**：趋势判断
- **英文**：Trend Judgment
- **补充**：对未来哪些能力会变轻、哪些能力会长期存在的判断

### 💬 一句话解释

说白了，随着模型变强，一些为了弥补旧模型短板的"临时补丁"会变轻；但真正负责组织系统的能力，不会消失。

### 🧐 你可以先这样理解

不是所有"模型外部"的东西都一样。可以分成两类：

| 类型 | 特征 | 未来趋势 |
| --- | --- | --- |
| 🩹 **临时补丁** | 只是因为旧模型不够强才存在 | 可能会逐渐变轻 |
| 🏗️ **系统层能力** | 无论模型多强都必须保留 | 会长期存在 |

**临时补丁的例子**：

- 过细的提示词技巧
- 很死的步骤拆分
- 大量的 少样本 示例

**系统层能力的例子**：

- 工具执行层（真正去操作外部系统）
- 权限、验证、审计
- 项目规则和业务约束

### ✅ 你只需要记住

- 🩹 不是所有外部约束都会消失
- 📉 更可能被删掉的，是那些只为弥补旧模型短板的临时补丁
- 🏗️ 更可能长期存在的，是那些负责承接模型能力、妥善组织的系统层能力

### 🧬 双螺旋：相辅相成的演进关系

AI 的发展不是单一路线的线性进步，而是**两条主线并行缠绕、相互促进**的双螺旋结构：

```
模型能力提升 ──────────────────────────────►
      ↑                                       ↓
      ↑  模型越强，对系统要求越高，推动系统进化  ↓
      ↑                                       ↓
      ←────────────────────────────── 系统能力发展
              系统越稳，越能释放模型潜力
```

| 阶段 | 模型派的发展 | 工程派的发展 | 相互关系 |
| :--- | :--- | :--- | :--- |
| 早期 | GPT-3 能生成流畅文本 | 提示工程、RAG 开始出现 | 模型能力有限，系统层以补丁为主 |
| 中期 | GPT-4 推理能力增强 | 工作流、Agent 框架成熟 | 模型变强，系统层可以更灵活 |
| 未来 | 模型更自主、更可靠 | 系统层转向生态和治理 | 两者边界模糊，深度融合 |

---

### 🎯 回到两个本质

| 本质 | 核心理解 |
| :--- | :--- |
| 🎯 **本质一（概率模型）** | 不管更强调模型还是系统，底层都建立在概率模型之上。大模型派推动模型本身处理更多，驾驭工程派在外部更好地组织模型输出——**两者共同拓展模型的能力边界**。 |
| 📋 **本质二（上下文）** | 不管哪条路，最终都绕不开模型上下文（Context）的组织方式。大模型派靠模型自己处理更长上下文，驾驭工程派靠外部系统补充和整理上下文——**两者共同扩大上下文范围**。 |

### 🌉 为下一章铺垫

明白了大模型派和驾驭工程派是相辅相成的双螺旋，下一个问题自然来了：

**智能体（Agent）这个产品形态，是怎么把这两条主线融合在一起的？** 它既需要强大的模型能力，也需要完善的系统设计——它是两条主线交汇的**结晶点**。

这就是下一章要聊的：**智能体是怎么"干活"的**——我们会看到，一个真正好用的智能体，必须同时做好模型能力和系统工程。
