# 第三章 AI 到底靠“脑子”还是靠“系统”？

## 读前先想

- 同样都在做 AI，为什么有人更相信模型，有人更相信系统？
- 前面那些技术放回这两条主线里看，到底差别在哪？

## 章节定位

| 维度 | 内容 |
| --- | --- |
| 横向定位 | 这是本项目解释两条主线的关键章节。 |
| 本章回扣 | 1. 不管更强调模型还是系统，底层都建立在概率模型之上。<br>2. 不管更强调模型还是系统，最终都绕不开模型上下文（Context）的组织方式。 |

## 先看两种思路

| 思路 | 它更看重什么 | 你可以先怎么理解 |
| --- | --- | --- |
| 大模型派（Big Model） | 模型本身是不是越来越强 | 更相信“脑子”会越来越强 |
| 驾驭工程派（Harness Engineering） | 模型外部的系统是不是足够稳 | 更相信“系统”要先把能力接住 |

## 第一节：为什么会有这两种做法

### 提问

- 为什么同样做 AI，最后会分出两种关注点？
- 为什么只谈模型或者只谈系统，都不够解释真实产品？
- 为什么这里要用“两条主线”这个框架来整理前两章内容？

### 术语

- 中文：两条主线
- 英文：two approaches
- 补充：这里指大模型派（Big Model）和驾驭工程派（Harness Engineering）两种不同的侧重点

### 一句话解释

AI 要真正解决问题，既需要模型能力进化，也需要外部系统把这股能力组织起来，所以真实产品里自然会出现两种不同的关注点。

### 你可以先这样理解

- 一条路更关心“模型本身能不能越来越强”。
- 另一条路更关心“模型外部的系统能不能把能力接住、管住、用起来”。
- 这不是为了故意分类，而是做产品时真的会遇到这两类问题。

### 示例

比如有人会先问：

`模型能不能自己把复杂任务一次性想明白？`

也有人会先问：

`就算模型会想，知识库、工具、权限和审计谁来接？`

这两个问题的出发点不同，就会自然走向两种不同的侧重。

### 你只需要记住

- 这不是纯理论争论，而是做产品时自然出现的两种思路。
- 一种更强调模型进化，一种更强调系统组织。

## 第二节：第一种思路，先把模型做强

### 提问

- 为什么随着模型越来越强，会有人想把更多事情直接交给模型本身？
- 为什么这一派会觉得很多外部补丁以后可能变轻？
- 为什么这里要用“先把脑子做强”这个白话框架来理解它？

### 术语

- 中文：大模型派
- 英文：Big Model
- 补充：更强调模型本身持续变强的思路

### 一句话解释

大模型派（Big Model）更相信，随着模型持续变强，很多原来需要额外补的能力，未来可以更多交给模型本身。

### 你可以先这样理解

- 如果模型理解能力、推理能力和长上下文处理能力不断增强，那么一些原来很依赖外部技巧的做法，可能会变轻，甚至被模型吸收。
- 这条思路更关心的是：模型本身能不能做得更多、更稳、更聪明。
- 它的基本判断是，很多复杂度最终可以往模型内部收。

### 示例

- 更强的模型，可能更容易直接理解复杂指令。
- 更强的模型，可能更会自己决定什么时候该调用工具。
- 更强的模型，可能让一些非常细碎的提示词技巧慢慢变得没那么重要。

### 你只需要记住

- 这条思路更看重模型能力本身的进化。
- 它更相信“脑子变强”会带来整体能力提升。

## 第三节：第二种思路，先把系统做稳

### 提问

- 为什么就算模型继续变强，很多团队还是会坚持先把系统做稳？
- 为什么真实工作里，模型本身不等于知识、工具、权限和流程？
- 为什么这里要用“先把系统做稳”这个框架来理解驾驭工程派？

### 术语

- 中文：驾驭工程派
- 英文：Harness Engineering
- 补充：更强调模型外部系统组织能力的思路

### 一句话解释

驾驭工程派（Harness Engineering）更相信，模型能力要真正落地，关键还在于模型外部的系统设计。

### 你可以先这样理解

- 模型本身不会自己接企业知识库，也不会自己真正执行工具。
- 模型本身也不等于权限控制、验证机制、审计流程和项目规则。
- 所以要让 AI 在真实工作里稳定做事，还需要外部系统把信息、工具、流程和约束组织起来。

### 示例

- 把公司文档接进来，是系统层的事情。
- 真正去发邮件、查日历、执行命令，也是系统层的事情。
- 让智能体（Agent）在高风险动作前停下来等人确认，也属于系统层的设计。

### 你只需要记住

- 这条思路更看重系统设计。
- 它更相信“系统做稳”是 AI 真正落地的关键。

## 第四节：用前两章的例子重新看一遍

### 提问

- 为什么前两章讲过的技术还要重新归类一遍？
- 为什么不能只记技术名字，而要看它更偏模型还是更偏系统？
- 为什么这里要用“两条主线”的框架重新看这些例子？

### 术语

- 中文：技术归类
- 英文：technology mapping
- 补充：把前两章出现过的技术重新放回两条主线中理解

### 一句话解释

把前两章的技术重新放回两条主线里看，读者才更容易看清它们各自更偏向在补“模型”还是在补“系统”。

### 你可以先这样理解

- 同一个技术名字，背后往往更偏向某一类问题。
- 如果不重新归类，读者容易把它们看成彼此孤立的小知识点。
- 一旦放回两条主线里，整套知识会更成体系。

### 对照表

| 例子 | 更偏向哪种思路 | 为什么 |
| --- | --- | --- |
| 检索增强生成（RAG） | 更偏向驾驭工程派 | 它通过外部检索给模型补资料 |
| 工具调用 / 函数调用（Tool Calling / Function Calling） | 更偏向驾驭工程派 | 真正执行工具的是外层应用，不是模型本身 |
| 工作流（Workflow） | 明显偏向驾驭工程派 | 它先用外部流程把步骤写死、管住 |
| 智能体（Agent） | 两边都重要 | 一边依赖更强模型，一边依赖更完整的系统组织 |
| 技能（Skill） | 更偏向驾驭工程派 | 它把常用做法提前整理成可复用的任务包 |

### 你只需要记住

- 前两章里的很多技术，并不是孤立存在的。
- 把它们放回这两条主线里看，整个项目会更容易理解。

## 第五节：我的判断，以后哪些会变轻，哪些会留下来

### 提问

- 为什么到了这里，还要继续追问“哪些会变轻，哪些会留下来”？
- 为什么不是所有外部系统能力都会被更强的模型直接吃掉？
- 为什么这里要从“临时补丁”和“长期系统能力”两个方向来看未来变化？

### 术语

- 中文：趋势判断
- 英文：trend judgment
- 补充：这里指对未来哪些能力会变轻、哪些能力会长期存在的判断

### 一句话解释

随着模型能力增强，一些只是为了弥补旧模型短板而存在的外部补丁，可能会变轻；但真正负责组织系统的那部分能力，不会消失。

### 你可以先这样理解

- 不是所有“模型外部”的东西都一样。
- 有些只是因为旧模型不够强，所以才额外补上的临时技巧。
- 也有些是无论模型多强，真实系统都必须长期保留的能力，比如权限、验证、审计和执行层。

### 表格

| 类型 | 更可能的变化 |
| --- | --- |
| 过细的提示词技巧 | 可能会逐渐变轻 |
| 很死的步骤拆分 | 可能会减少 |
| 工具执行层 | 仍然会长期存在 |
| 权限、验证、审计 | 仍然会长期存在 |
| 项目规则和业务约束 | 仍然会长期存在 |

### 你只需要记住

- 不是所有外部约束都会消失。
- 更可能被删掉的，是那些只为弥补旧模型短板的临时补丁。
- 更可能长期存在的，是负责把模型能力接住、管住、用起来的系统层能力。
