# 🚀 第二章 从 ChatGPT 到智能体：AI 应用是怎么一步步进化的？

## 🤔 读前先想

> 💭 在往下看之前，不妨先想想：

- 从 ChatGPT 只能"聊"，到今天 AI 能"做"，中间差了什么？
- 为什么 AI 不是一开始就能"做事"，而是要从 ChatGPT 这个阶段一步步走过来？

---

## 🧭 章节定位

| 维度 | 内容 |
| :--- | :--- |
| 📍 **横向定位** | 这一章从产品演进的角度，解释 AI 为什么会不断补信息、动作和流程。 |
| 🎯 **本章回扣** | 1️⃣ 回到本质一：大模型始终是生成文字的概率模型。<br>2️⃣ 回到本质二：所有演进本质上都在不断扩大模型上下文（Context）的范围。 |

## 📊 阶段总览

> 💡 **这不是人为划分的阶段，而是真实产品演进中逐步出现的能力。**

| 阶段 | 产品形式 | 上一阶段哪里不够 | 新增能力 | 本质回扣 |
| :--- | :--- | :--- | :--- | :--- |
| 💬 ChatGPT 时刻 | 在网页中一问一答 | 普通人终于能直接对话，但模型只能看眼前对话 | 让普通人直接和模型交流 | 上下文范围：当前对话 |
| 🔍 知识增强阶段 | 先查资料再回答 | 参数知识有限且会过时 | 检索增强生成（RAG） | 上下文范围：+ 外部知识库 |
| 🛠️ 工具调用阶段 | 回答之外还能触发动作 | 只会说，不会真的做事 | 工具调用 | 上下文范围：+ 工具说明和执行结果 |
| 📋 工作流阶段 | 多个步骤按顺序串起来 | 工具多了，但模型稳定性还不够 | 外部流程编排 | 上下文范围：+ 多步骤的中间结果 |
| 🤖 智能体阶段 | 围绕目标连续执行 | 固定流程不够灵活 | 多步决策和动态调整 | 上下文范围：+ 目标、规划、反思等元信息 |

> 🎯 **一句话总结**：五个阶段的演进，表面看是能力越来越强，本质上是**上下文的范围越来越大**。

---

## 📖 第一小节：ChatGPT 时刻——一切从"能聊"开始

### ❓ 提问

- 为什么 ChatGPT 成了 AI 时代的"开门时刻"？
- 为什么它一开始只能"聊"，不能直接帮我订机票？

---

### 📝 术语

| 项目 | 内容 |
| :--- | :--- |
| **中文** | 网页聊天机器人 |
| **英文** | Web Chatbot |
| **补充** | ChatGPT 让这种形式真正进入大众视野 |

---

### 💬 一句话解释

> 💡 **说白了，ChatGPT 开启了"对话即界面"的时代——你打字，它回话，但也就仅限于此了。**

---

### 🧐 你可以先这样理解

2022 年底，ChatGPT 突然火了。这不是一个简单的"新产品上线"，而是一个**时代的开端** 🚀。

无数人第一次在网页里输入问题，然后看着 AI 一句一句地回复。那种感觉就像突然有了一个"什么都能聊"的伙伴 💬。学生用它写论文，程序员用它 debug，职场人让它帮忙写邮件……它似乎什么都能聊，但又好像什么都做不了——你不能真的让它去查你的日程、订机票、或者操作公司的系统。

这一阶段最重要的变化，其实是**普通人第一次可以直接和大模型对话**。但它的底层并没有变：它仍然是根据当前对话里的上下文，按概率生成文字。它能依赖的信息，主要还是训练时学到的参数知识，以及你眼前这段对话。

换句话说，它看起来是在"聊天"，但实际上还是在做第一章讲的那件事：**根据前文，猜下一个词**。

**❓ 那为什么不让它直接做事呢？** 

因为那时候它连"当前对话"以外的信息都看不到，更别说去操作外部系统了。

---

### 📚 示例

**✅ 如果你问：**

```
请用白话解释一下什么是周报
```

它通常能回答得不错。

**❌ 但如果你问：**

```
请告诉我我们公司现在最新的报销制度
```

它就未必知道，因为这类信息不在它训练时的参数里，也没有被放进当前的上下文。

---

### ✅ 你只需要记住

| 要点 | 说明 |
| :--- | :--- |
| 🏁 **起点** | 最早的大模型产品，最典型的形式是网页聊天机器人。 |
| 🧠 **本质** | 它看起来像在聊天，但底层还是生成文字的概率模型。 |
| ⚠️ **限制** | 这一阶段最大的限制，本质上是可用的上下文不够——只能看到当前对话。 |

---

💭 **上一节我们回顾了 ChatGPT 时刻**：那时候的 AI 只能看到当前对话，所以答不上公司的新制度。那怎么办？答案很简单——帮它查资料。

## 📖 第二小节：知识增强阶段（RAG）

### ❓ 提问

- 大模型明明"学富五车"，为什么却答不上公司的新制度？
- 既然参数知识不够，那怎么让模型"看到"它没学过的东西？

---

### 📝 术语

| 项目 | 内容 |
| :--- | :--- |
| **中文** | 检索增强生成 |
| **英文** | Retrieval-Augmented Generation |
| **简称** | RAG |
| **补充** | 先查资料，再让模型回答 |

---

### 💬 一句话解释

> 💡 **说白了，RAG 就是先帮模型"查资料"，把查到的内容塞进上下文，再让它基于这些新信息回答。**

---

### 🧐 你可以先这样理解

上一节我们说了，模型只能看到当前对话，所以答不上公司的新制度。怎么办？

答案很简单：**帮它查资料，然后把资料放进上下文** 📚。

参数里的知识有限，而且会过时。真实场景里，很多关键信息在网页、知识库、公司文档和课程资料里。RAG 做的事情，就是在用户提问后，系统先去检索相关资料，再把资料放进上下文，让模型基于这些资料继续生成答案。

> 📈 **上下文范围从这里开始扩大了**——从"只能看到对话"，变成了"还能看到检索到的资料"。

---

### 🔄 流程图

```mermaid
flowchart LR
    A["💬 用户问题"] --> B["🔍 检索系统"]
    B --> C["📚 知识库 / 文档"]
    C --> D["📄 取回相关资料"]
    D --> E["📋 放入上下文（Context）"]
    E --> F["🤖 大模型生成答案"]
```

---

### 📚 示例

比如你问：

```
根据我们团队的请假制度，帮我写一条请假申请
```

如果模型没有看到制度文档，它可能只能泛泛地写。

但如果系统先从知识库里检索出请假制度，再把这段资料放进上下文，它通常就能写得更贴近你们团队的真实要求。

---

### ✅ 你只需要记住

| 要点 | 说明 |
| :--- | :--- |
| 🎯 **本质** | RAG 不是让模型突然变聪明。 |
| 🔧 **做法** | RAG 做的事情，本质上是给模型补充更相关的上下文。 |
| 📈 **扩展** | 上下文范围从这里开始扩大：从只能看到对话，变成了还能看到检索到的资料。 |

---

💭 **前两节解决了"知道更多"的问题**：第1节能对话，第2节能查资料。但下一个问题是——很多任务不是"回答一句话"就结束了，而是要真的去做事。AI 怎么才能真的"动手"？

## 📖 第三小节：工具调用阶段

### ❓ 提问

- AI 只会"说"还不够，怎么让它真的"做"？
- 为什么 AI 不能直接动手，非要通过"表达意图"这种拐弯的方式？

---

### 📝 术语

| 项目 | 内容 |
| :--- | :--- |
| **中文** | 工具调用 / 函数调用 |
| **英文** | Tool Calling / Function Calling |
| **补充** | 模型表达意图，外层系统负责执行 |

---

### 💬 一句话解释

> 💡 **说白了，工具调用就是让模型学会"发号施令"——它告诉你该调用什么工具、传什么参数，但真正动手的是外层系统。**

---

### 🧐 你可以先这样理解

很多任务不是"回答一句话"就结束了，而是要真的去做事。

但到了这一步，我们终于明白一件事：**大模型本质上是生成文字的概率模型，它不能直接操作外部世界。** 它既不能真的点开日历，也不能真的发送邮件。那怎么办？

办法是：**让模型"表达意图"** 🎯。

系统先把工具的说明（功能、参数、返回值）放进上下文，模型读完后，输出"我要调用哪个工具、参数是什么"的意图；外层应用执行后，再把结果放回上下文，模型基于新结果继续生成。

> 📈 **上下文范围再次扩大**——从"能看到对话+资料"，变成了"还能看到工具说明和执行结果"。

---

### 🔄 流程图

```mermaid
flowchart LR
    A["🎯 用户目标"] --> B["📋 工具说明进入上下文（Context）"]
    B --> C["🤖 大模型输出调用意图"]
    C --> D["⚙️ 外层应用执行工具"]
    D --> E["📋 执行结果回到上下文（Context）"]
    E --> F["🤖 大模型继续生成下一步内容"]
```

---

### 📚 示例

比如你说：

```
帮我把今天下午三点的组会记进日历，并提醒我提前十分钟准备
```

模型不会真的自己去点开日历。

它会先根据工具说明，输出调用日历工具的意图。然后外层应用去真正创建日程，再把结果返回给模型。

---

### ✅ 你只需要记住

| 要点 | 说明 |
| :--- | :--- |
| 👔 **角色** | 工具调用时，大模型更像决策层，不是执行层。 |
| 🗣️ **方式** | 它不是在亲手调用工具，而是在表达调用意图。 |
| 📈 **扩展** | 上下文范围再次扩大：从能看到对话+资料，变成了还能看到工具说明和执行结果。 |

---

### 🌉 补充：MCP——让工具连接更标准化

> 💡 **一句话理解**：MCP 就是给"工具怎么接"定个标准，让模型和外部工具的连接更规范、更可复用。

当工具越来越多、接入方式五花八门时，一个问题浮现了：**每个工具都要单独对接吗？**

MCP（模型上下文协议，Model Context Protocol）就是解决这个问题的一个标准化尝试。它定义了一套通用的连接规范，让外部工具和数据源可以按照统一的方式接入，模型也能用统一的方式来发现和调用它们。

**MCP 的核心价值**：
- 🔌 **标准化接口**：不用为每个工具写不同的接入代码
- 📦 **可复用连接**：同一个 MCP 服务，可以被不同的 AI 应用共用
- 🏗️ **生态建设**：工具提供者按标准开发，AI 应用按标准接入

**但要注意**：MCP 是 2024 年底才出现的协议标准，它不是在"工具调用"阶段就有的东西，而是后来为了让工具连接更规范、更规模化而出现的一层标准化设计。

---

💭 **第3节（加上 MCP 补充）解决了"能动手"和"怎么标准连接"的问题**，但新问题又来了：模型虽然能决定调用什么工具，但它不够稳——有时候该调 A 却调了 B，有时候多调或少调了步骤。怎么办？

## 📖 第四小节：工作流阶段

### ❓ 提问

- 工具接进来了，为什么还需要"流程"来管？
- 既然模型能自己决定调用工具，为什么还要人工写死流程？

---

### 📝 术语

| 项目 | 内容 |
| :--- | :--- |
| **中文** | 工作流 |
| **英文** | Workflow |
| **补充** | 把多个固定步骤预先串起来的一套外部流程 |

---

### 💬 一句话解释

> 💡 **说白了，工作流就是"先让模型在轨道上跑"——因为那时候模型还不够稳，容易跑偏，所以人先把步骤写死。**

---

### 🧐 你可以先这样理解

工具调用出现后，新问题又来了：模型虽然能决定调用什么工具，但它不够稳。有时候该调 A 却调了 B，有时候多调或少调了步骤。

所以这个阶段的做法是：**在模型外部先把步骤写死** 📋。

外层系统会提前规定先做什么、后做什么、哪一步用什么工具、结果怎么往下传。

这样做的好处是稳定、可控。但代价也很明显：灵活性不足，遇到流程外的情况容易卡住。

> 📈 **上下文范围继续扩大**——从单步的工具调用，变成了多步骤的中间结果和流程状态。

---

### 📊 工作流阶段特点

| 维度 | 工作流阶段的表现 |
| :--- | :--- |
| 🤔 **为什么会出现** | 模型当时的自主性和稳定性还不够强 |
| ✅ **优势** | 稳定、可控、容易复用 |
| ⚠️ **局限** | 灵活性不足，流程外情况容易卡住 |
| 🎯 **本质回扣** | 上下文范围扩大：+ 多步骤的中间结果 |

---

### 📚 示例

比如一个常见流程是：

```
先读取会议纪要 -> 再提取待办事项 -> 再整理成周报草稿 -> 最后发给负责人确认
```

这个流程在重复任务里很好用，因为每一步都提前设定好了。

但如果中间突然出现新的目标，或者资料格式和预设不一样，工作流就可能接不住。

---

### ✅ 你只需要记住

| 要点 | 说明 |
| :--- | :--- |
| 🤔 **原因** | 工作流阶段出现，是因为模型当时还不够强，需要外部流程来约束。 |
| ⚖️ **权衡** | 优势是稳定、可控、容易复用；局限是灵活性不够。 |
| 📈 **扩展** | 上下文范围继续扩大：从单步的工具调用，变成了多步骤的中间结果。 |

---

💭 **第4节用固定流程解决了"稳定性"的问题**，但代价是灵活性不足。如果模型变强了，是不是可以给它更多自主权？这就来到了智能体阶段。

## 📖 第五小节：智能体（Agent）阶段

### ❓ 提问

- 固定流程不够用了，AI 怎么变得更"灵活"？
- 那模型变强了，是不是就不需要外部流程了？

---

### 📝 术语

| 项目 | 内容 |
| :--- | :--- |
| **中文** | 智能体 |
| **英文** | Agent |
| **补充** | 围绕一个目标持续完成多步任务的 AI 应用 |

---

### 💬 一句话解释

> 💡 **说白了，智能体（Agent）就是让模型在更大的范围内"自主发挥"——它不再只是按固定步骤走，而是围绕目标自己决定怎么做。**

---

### 🧐 你可以先这样理解

到了智能体阶段，AI 不再只是答一句或执行固定流程，而是**围绕一个目标连续工作** 🎯。

模型能力变强后，原来必须写死的部分步骤，可以交给模型自己做局部决策。但底层还是同一个概率模型，只是外层系统把目标、工具、结果和更长的上下文组织得更完整了。

> 📈 **上下文范围扩大到最大**——除了对话、资料、工具、中间结果，还包括目标、规划、反思等元信息。模型基于这些更丰富的上下文，来决定下一步做什么。

---

### 🎒 关于"技能（Skill）"

智能体不是凭空就能做所有事的。为了让它能处理特定类型的任务，系统通常会提供**技能（Skill）**——可以理解成一种"能力包"，把某类任务常用的说明、工具入口和执行规则提前整理好。

技能本身不是新的大脑，也不是替代模型。它的作用是在需要时，把一类任务的说明和约束更稳定地放进上下文，让模型知道"这类任务通常怎么做"。

---

### 📊 工作流 vs 智能体对比

| 维度 | 📋 工作流 | 🤖 智能体（Agent） |
| :--- | :--- | :--- |
| **步骤安排** | 主要靠人工预先写死 | 可以由模型参与动态调整 |
| **适合任务** | 重复、固定、边界清楚 | 更复杂、更开放、变化更多 |
| **好处** | 稳定、可控 | 更灵活，更能处理非标准任务 |
| **主要风险** | 流程僵化 | 多步任务更容易跑偏，权限和成本更难控制 |
| **本质回扣** | 上下文范围：多步骤的中间结果 | 上下文范围：+ 目标、规划、反思等元信息 |

---

### 📚 示例

比如你说：

```
帮我准备明天的课程展示：先整理资料，再列一个提纲，最后给我一版三分钟发言稿
```

如果只是普通聊天机器人，它更可能只给你一段泛泛的回答。

但智能体形态的系统，会更像是在围绕这个目标连续工作，把不同步骤的结果继续放回上下文，再推进下一步。

---

### ✅ 你只需要记住

| 要点 | 说明 |
| :--- | :--- |
| 🧬 **本质** | 智能体（Agent）不是另一种全新的东西，而是在第一章两个核心点之上继续发展出来的应用形态。 |
| 🎯 **核心** | 它底层仍然是生成文字的概率模型，外层系统把上下文组织得更完整。 |
| 📈 **扩展** | 上下文范围最大：除了对话、资料、工具、中间结果，还包括目标、规划、反思等元信息。 |

---

## 📝 本章要义

> 💡 **一句话核心结论**：AI 产品从"能聊"到"能做"的演进，本质上是**上下文范围不断扩大**的过程——模型始终是那个"接话"的概率模型，只是我们能帮它把"接话的依据"给得越来越充分。

| 📖 第3节 | 🛠️ 工具调用 | 只会说，不会真的做事 | + 工具说明和执行结果 |
| 📖 第4节 | 📋 工作流 | 模型稳定性还不够 | + 多步骤的中间结果 |
| 📖 第5节 | 🤖 智能体（Agent） | 固定流程不够灵活 | + 目标、规划、反思等元信息 |

---

### 🎯 回到两个本质

| 本质 | 核心理解 |
| :--- | :--- |
| 🎯 **本质一（概率模型）** | 无论哪个阶段，底层都是那个按概率生成文字的大模型。它不是在"变聪明"，只是能看到的上下文越来越多。 |
| 📋 **本质二（上下文）** | 五个阶段的演进，表面是能力越来越强，本质是上下文范围不断扩大——从只能看到对话，到能看到知识库、工具、流程状态和元信息。 |
