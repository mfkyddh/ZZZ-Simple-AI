## 📖 第一小节：选工具——别选最强的，选最配的

### ❓ 提问

> 市面上AI工具这么多，我怎么知道哪个适合我？  
> 是不是选功能最全、模型最强的就对了？

### 💬 一句话理解

> 选工具不是选"最强"的，是选"最配你场景"的。

### 🧐 你可以先这样理解

想象你要买一双鞋：
- 跑步鞋很强，但如果你要爬山，不如登山鞋合适
- 六部件框架就是你的"试鞋清单"——帮你判断这双鞋合不合脚

用六部件评估一个AI工具，问自己这六个问题：

| 问题 | 为什么重要 |
|:---|:---|
| 它用的什么模型？适合我的任务吗？ | 复杂任务需要强模型，简单任务普通模型就够 |
| 它能记住多少上下文？ | 长文档分析需要大上下文，日常聊天不需要 |
| 它能调用我需要的工具吗？ | 办公场景需要文档/邮件工具，编程场景需要代码工具 |
| 错了怎么办？有确认机制吗？ | 高风险任务需要严格边界，低风险任务可以更开放 |
| 我怎么知道做得对不对？ | 需要明确验收标准才能判断输出质量 |
| 工作节奏适合我吗？ | 紧急任务要快，复杂任务要稳 |

**不是看哪个分数高，是看哪个在你最在乎的维度上得分高。**

### 📚 快速决策法

| 如果你主要做... | 重点看什么 | 选择方向 |
|:---|:---|:---|
| **一次性简单任务** | 上手快不快 | 通义千问、豆包、DeepSeek |
| **重复性工作** | 能不能保存套路 | 支持Skill的工具 |
| **高风险任务** | 有没有确认机制 | Claude Code、CodeBuddy |
| **编程开发** | 工具全不全 | Cursor、Trae、Claude Code |

### ✅ 你只需要记住

1. **没有最好的工具，只有最合适的工具**
2. **用六部件框架快速评估，别被宣传语迷惑**
3. **先试用，再决定——实践出真知**

## 📖 第二小节：用工具——出问题先检查这三样

### ❓ 提问

> AI输出不对，是模型不行，还是我用得不对？  
> 怎么排查问题出在哪？

### 💬 一句话理解

> 80%的问题不出在模型，出在你没把任务说清楚——先检查目标、上下文、边界。

### 🧐 你可以先这样理解

AI就像一位新入职的实习生：
- 你说"写个报告"——他不知道写什么报告、给谁看、多长
- 你说"写个报告"，加上"关于Q3销售数据的，给总经理看的，一页纸"——他就明白了

**三板斧排查法**：

**第一板斧：目标说清楚了吗？**
```
❌ 模糊："分析这个数据"
✅ 清楚："分析这个销售数据，找出Q3比Q2下降的原因"
```

**第二板斧：上下文给够了吗？**
```
❌ 不足：直接问"怎么改进？"
✅ 充分："背景是...目标是...限制条件是..."
```

**第三板斧：边界设了吗？**
```
❌ 开放："随便写"
✅ 约束："500字以内，用大白话，别说专业术语"
```

### 📚 常见问题和排查

| 问题表现 | 最可能原因 | 怎么解决 |
|:---|:---|:---|
| AI理解错了方向 | 目标没说清 | 用"你要做的是...目的是..."重新描述 |
| AI漏了重要信息 | 上下文不够 | 把相关背景、数据、文件传给它 |
| AI输出太长/太短 | 边界没设 | 明确字数、格式、风格要求 |
| AI用了错误的方法 | 工具说明不清 | 告诉它"用XX方法，不要YY方法" |

### ✅ 你只需要记住

1. **先检查自己的输入，再怀疑模型能力**
2. **目标、上下文、边界——这三样搞定，80%问题没了**
3. **从小任务开始，逐步积累经验**

## 📖 第三小节：沉淀经验——从每次教到一次封装

### ❓ 提问

> 为什么我教AI做一件事，下次还要重新教？  
> 有没有办法让它记住我的套路？

### 📝 术语

- **Skill**：任务目标 + 上下文模板 + 执行步骤 + 验收标准的封装

### 💬 一句话理解

> Skill就是把"你怎么教AI做这件事"的经验，保存下来下次直接用。

### 🧐 你可以先这样理解

想象你每周都要写周报：
- **第一次**：告诉AI"帮我写周报，包含本周完成、下周计划、遇到的问题"
- **第二次**：又要说一遍...
- **第N次**：每次重复，很烦

Skill就是把这个过程**封装**下来：
- 固定格式是什么
- 需要哪些输入信息
- 输出应该长什么样
- 怎么算好、怎么算不好

封装好以后，以后只说"用周报Skill"，AI就知道整套流程。

### 📚 Skill封装实战：周报生成器

**Step 1：观察和提炼套路**

你每次写周报都要：
1. 回顾本周做了什么事
2. 整理数据/成果
3. 写下周计划
4. 列出遇到的问题

**Step 2：结构化描述**

```
【Skill名称】周报生成器

【任务目标】根据本周工作数据，生成符合公司格式的周报

【需要输入】
- 本周完成事项清单
- 关键数据/成果
- 遇到的问题

【执行步骤】
1. 按"完成事项→数据亮点→遇到的问题"组织内容
2. 每项工作配一句简要说明
3. 写3条下周计划

【输出格式】
【本周完成】
- xxx
【数据亮点】
- xxx  
【遇到问题】
- xxx
【下周计划】
- xxx

【验收标准】
- 一页纸以内
- 用大白话，不要专业术语
- 数据要突出，问题要诚实
```

**Step 3：让AI帮你自动封装**

现在的AI工具越来越智能——**你不需要手动写Skill模板，只要带着AI走一遍，它就能自动总结、形成标准化的Skill**。

**以Claude Code为例**：
```
你：帮我写周报，要包含本周完成、下周计划、遇到的问题
（Claude生成初稿）

你：不对，格式应该是...
（Claude调整）

你：还有，下次记得先问我要数据，别自己编
（Claude记录偏好）

Claude：我总结了这个流程，包含目标、输入、步骤和格式。
        要保存为Skill吗？以后可以直接调用。
        
你：保存，叫"周报生成器"
```

Claude Code会自动生成一个标准化的Skill文件，包含：
- 触发条件（什么时候用）
- 需要的信息（输入）
- 执行步骤（怎么做）
- 输出格式（长什么样）

**其他工具类似**：
- Cursor：通过 `.cursorrules` 或对话让AI总结
- 通义千问：在智能体市场中创建个人智能体
- 大部分AI工具：多轮对话后，AI会提示"是否保存为模板/Skill"

### ✅ 你只需要记住

1. **Skill = 你的经验 + AI的执行能力**
2. **不需要手写模板——带着AI走一遍，它自动帮你封装**
3. **从重复最多的任务开始，边用边沉淀**

## 📖 第四小节：站在巨人的肩膀上——先用现成的 Skill

### ❓ 提问

> 每次都要自己封装Skill吗？  
> 有没有别人已经封装好的可以直接用？

### 💬 一句话理解

> 大多数常见任务，早就有人封装好了——先找现成的，没有才自己造。

### 🧐 你可以先这样理解

Skill生态正在快速发展，就像手机App商店一样：
- **通用任务**：写周报、回邮件、整理数据...早就有成熟方案
- **行业专用**：代码审查、财务分析、客服话术...各个领域都有
- **个人定制**：只有你自己独特的 workflow 才需要自己封装

**为什么要先用现成的？**
1. **省时间**：不用从零摸索
2. **质量高**：别人已经迭代优化过
3. **学思路**：看看别人怎么封装，学习借鉴

### 📚 主流AI工具的Skill生态

| 工具 | Skill生态 | 如何使用别人的Skill |
|:---|:---|:---|
| **Claude** | Claude Skills Store | 浏览市场 → 点击"添加" → 直接调用 |
| **Cursor** | Cursor Rules + Templates | 导入 `.cursorrules` 文件或从社区下载 |
| **通义千问** | 智能体市场 | 搜索场景 → 添加智能体 → 一键使用 |
| **Coze/扣子** | 插件+工作流市场 | 浏览插件 → 添加到自己的Bot |

**使用方式**：找到合适的Skill → 导入/添加 → 直接调用，不用从零开始

### 📚 Skill的复用方式

**方式一：使用"导入Skill"功能**

大部分AI工具支持导入外部Skill：
```
1. 从Skill市场/社区下载Skill文件（通常是 .json/.md/.yaml 格式）
2. 在AI工具中选择"导入Skill"/"添加自定义Skill"
3. 选择文件，完成导入
4. 以后直接调用这个Skill
```

**方式二：放入通用目录，多工具共享**

更高级的方式——把Skill文件放在所有AI工具都能访问的位置：
```
项目目录
└── .agents/
    └── skills/
        ├── 周报生成器/
        │   └── SKILL.md
        ├── 邮件回复助手/
        │   └── SKILL.md
        └── 代码审查/
            └── SKILL.md
```

**支持 `.agents` 目录的工具**：
- **OpenCode**：原生支持 `.agents/skills/`
- **Claude Code**：兼容 `.agents/skills/`
- **Codex**：遵循 Agent Skills 开放标准

**好处**：
- 一次配置，多工具通用
- 团队协作时，共享同一个Skill库
- 备份方便，换工具时Skill不丢失

### 📚 找不到现成的怎么办？

如果市场上没有合适的Skill，就自己封装。方法在第3节讲过：**带着AI走一遍，它自动帮你总结形成标准化Skill**。

### 📚 选用 vs 自建的决策

| 场景 | 建议 | 原因 |
|:---|:---|:---|
| **常见通用任务**（周报、邮件、翻译） | **先用现成的** | 别人优化过，质量有保障 |
| **行业专用任务**（代码审查、财务分析） | **找行业Skill** | 可能有人分享过类似需求 |
| **个人独特 workflow** | **自己封装** | 只有你自己知道要什么 |
| **敏感数据/特殊要求** | **自己封装** | 控制权和隐私更有保障 |

### ✅ 你只需要记住

1. **大多数任务都有现成的Skill——先找再用**
2. **带着AI走一遍，它能自动提炼Skill**
3. **从别人的Skill里学习，再优化成自己的**
